Linux总复习

第一讲

起源

Linux的起源可以追溯到1991年，当时一位名叫Linus Torvalds的芬兰大学生开始为自己的个人电脑开发一种新的操作系统。这个系统最初是为了满足他在学校计算机课程中的需求而设计的。在他的学习和开发过程中，他使用了Minix操作系统的代码和想法，并结合了自己的创新，最终开发出了一种新的操作系统，也就是现在所知道的Linux。

历史

  1. 1991年-1993年：最初的开发阶段 在1991年，Linus Torvalds开始开发Linux操作系统的内核，并在1992年发布了第一个可用版本。此时Linux还只是一个小型项目，只有几千行代码。在这个阶段，Linux主要用于个人电脑和服务器领域。
  2. 1994年-1999年：Linux的商业化和扩展 在1994年，Red Hat公司成立，开始将Linux作为商业产品销售。同时，其他公司也开始对Linux进行开发和支持，包括IBM、Novell和Oracle等。在这个阶段，Linux不断扩展，支持更多的硬件和应用程序，并逐渐进入企业市场。
  3. 2000年-2007年：Linux的稳定性和成熟度提高 在2000年代初，Linux开始受到更多企业的青睐，并逐渐被用于一些重要的应用场景，如金融、医疗和航空等领域。此时，Linux的稳定性和成熟度得到了极大的提高，并且逐渐成为企业级操作系统的首选。
  4. 2008年-至今：Linux的广泛应用和开源生态系统的壮大 在2008年之后，Linux得到了更广泛的应用，包括移动设备、嵌入式系统和云计算等领域。同时，Linux的开源生态系统也逐渐壮大，包括各种开源软件和工具，如Docker、Kubernetes、OpenStack等。这些开源技术对现代计算产生了重要的影响，并且Linux已经成为许多企业和组织的核心技术。

特点

  1. 开放性（遵循标准）

    Linux是一种开源的操作系统，任何人都可以查看、修改、使用和分发它的源代码。这意味着Linux的用户可以自由地定制和配置它，以适应他们的需求。

  2. 多用户、多任务

    Linux可以支持多个用户同时登录和使用同一台计算机，并且可以在后台同时运行多个程序和任务，不会因为资源竞争而出现系统崩溃。

  3. 良好的用户界面

  4. 设备独立性（把外部设备当作文件处理）

    Linux的一个重要特点是设备独立性，它能够把外部设备（如硬盘、光驱、打印机、串口、USB设备等）当作文件处理。这意味着Linux把这些设备看做是文件系统中的一个文件，用户可以使用标准的文件操作函数来访问这些设备，而不需要了解它们的具体硬件细节。

    在Linux中，每个设备都有一个对应的设备文件，这些文件被存储在/dev目录下，用户可以通过这些设备文件来访问和操作设备。例如，硬盘的设备文件为/dev/hd（其中表示硬盘的分区），打印机的设备文件为/dev/lp（其中表示打印机的编号）等。

    通过这种设备独立性的机制，Linux操作系统能够兼容多种不同的外部设备，并且可以很容易地进行扩展和升级。另外，由于外部设备被抽象为文件，因此用户可以使用标准的文件操作函数来进行读写、复制、移动、重命名等操作，这大大简化了对设备的操作和管理。

  5. 丰富的网络功能

    • 网络协议支持：Linux支持多种网络协议，包括TCP/IP、UDP、ICMP、HTTP、FTP、SSH等，这些协议使得Linux能够在各种网络环境中进行通信和数据传输。
    • 路由和网络管理：Linux提供了强大的路由和网络管理功能，使得管理员可以轻松地管理和监控网络连接、带宽、流量、安全等方面。
    • 网络安全：Linux具有多层次的网络安全功能，包括防火墙、网络认证、加密和虚拟专用网络（VPN）等，能够保障网络的安全性和可靠性。
    • 服务器和服务：Linux可以作为一个网络服务器，提供各种服务，包括Web服务器、邮件服务器、文件服务器、数据库服务器、DNS服务器等，这些服务能够满足不同用户的需求。
    • 网络性能优化：Linux提供了丰富的网络性能优化功能，包括内核优化、网络调整、缓存机制等，可以提高网络的速度和稳定性。

  6. 可靠的系统安全

    Linux是一个非常稳定和可靠的操作系统，通常不会出现崩溃或死机等问题。同时，Linux也被认为是一个相对安全的操作系统，因为它的内核可以隔离和保护用户的数据和程序。

  7. 良好的可移植性

    Linux可以运行在各种不同的硬件和平台上，包括个人电脑、服务器、嵌入式设备和移动设备等。同时，许多Linux应用程序也可以在不同的平台之间移植和运行。

GNU GPL

GNU GPL（GNU General Public License）是一种自由软件许可证，是由自由软件基金会（Free Software Foundation，FSF）发起的。该许可证规定，任何使用GNU GPL授权的软件必须在其源代码公开发布的同时，同样采用GNU GPL许可证，也就是说，这些软件必须是开源软件，并且允许其他人自由地使用、修改和再分发。GNU GPL还规定，任何使用该软件的人必须提供原始的源代码和版权声明，并在修改后的软件中明确说明。

GNU GPL在开源软件界中具有广泛的应用，许多著名的开源软件和操作系统，如Linux操作系统、GNU工具集、MySQL数据库等，都采用了GNU GPL许可证。此外，许多开源社区和组织也采用GNU GPL作为其开源软件的许可证，以保障开源软件的自由和开放性。

总的来说，GNU GPL是一种为保护自由软件而设计的开源软件许可证，它旨在鼓励开源社区的自由合作、共享和创新，使得开源软件的生态系统更加健康、稳定和可持续。

常见的发行版本

  1. Ubuntu：一款非常流行的Linux发行版本，以易用性和友好的界面而著称，适用于桌面和服务器环境。
  2. Debian：一个稳定、可靠的Linux发行版本，以其强大的软件包管理系统和安全性而著称，适用于服务器和桌面环境。
  3. CentOS：一款基于Red Hat Enterprise Linux（RHEL）源代码的社区发行版本，以其稳定性和安全性而著称，适用于服务器环境。
  4. Fedora：一个面向开发者和技术爱好者的Linux发行版本，以其先进的软件包管理系统和最新的技术特性而著称，适用于桌面和服务器环境。
  5. Arch Linux：一个自由、轻量级的Linux发行版本，以其极简主义和灵活性而著称，适用于技术爱好者和高级用户。
  6. openSUSE：一个稳定、易用的Linux发行版本，以其强大的YAST系统配置工具和开放性而著称，适用于桌面和服务器环境。
  7. Kali Linux：一款专门用于网络安全和渗透测试的Linux发行版本，以其预装了大量安全工具而著称，适用于安全测试人员和渗透测试爱好者。
  8. Red Hat Enterprise Linux（RHEL）：这是一款面向企业级用户的Linux发行版本，以其高度的稳定性、安全性和可靠性而著称。RHEL提供了一系列的服务和工具，如软件包管理、安全更新、系统配置、集成虚拟化等，可以满足企业级用户对于Linux操作系统的各种需求。

商业、共享、自由和免费软件的区别和联系

  1. 商业软件：商业软件是指由商业公司或组织开发并销售的软件。商业软件通常需要付费购买或使用，其源代码不公开，用户无权对其进行修改或分发。商业软件通常具有稳定性和专业性，适合用于商业、工业和科学领域等。
  2. 共享软件：共享软件是指由开发者免费提供、允许复制和分发的软件。共享软件可以是开源或闭源的，通常允许用户对其进行修改和定制，但需要遵守软件许可协议中的规定。共享软件的目的是鼓励创新和合作，推动知识和技术的自由流通。
  3. 自由软件：自由软件是指用户可以自由使用、复制、修改和分发的软件。自由软件的源代码必须公开，用户有权修改和改进该软件，并且需要将改进后的版本公开发布。自由软件的目的是保护用户的自由和隐私，促进软件的互联互通和技术的共享。
  4. 免费软件：免费软件是指用户可以免费获取和使用的软件，其源代码可以是公开或闭源的，用户无权修改和分发该软件。免费软件的目的是提供一个免费的替代品，以满足用户的一些基本需求，如文字处理、绘图等。

Linux目录结构

Linux以目录的方式来组织和管理系统中的所有文件，以根目录 / 为起点，所有其他的目录都由根目录派生而来，采用了一种层次化的树形结构，称为文件系统层次结构（Filesystem Hierarchy Standard，FHS）。

常见目录的作用和存放内容

  1. /bin：常用系统程序目录
  2. /boot：开机设定目录，也是存放核心 vmlinuz 的地方
  3. /dev：存放系统设备装置文件的目录
  4. /etc：系统配置文件，尤其 passwd, shadow
  5. /etc/rc.d/init.d：存放系统开机时载入服务的脚本
  6. /home：系统使用者的目录
  7. /lib：Linux 执行或编译程序函数库目录
  8. /mnt：软驱与光驱接入挂载的地方
  9. /proc：系统核心与执行程序的一些信息
  10. /root：系统管理员的目录
  11. /usr/bin, /bin：一般执行文件摆放的地方
  12. /usr/sbin, /sbin：系统管理员常用指令集
  13. /var：摆放系统日志文件的地方
  14. /lost+fount：摆放系统不正常产生错误时遗失的片段

Linux系统结构

分为用户空间和内核空间。

用户空间有应用程序、GUN C函数库。

内核空间有系统调用接口、内核、依赖于体系结构的内核代码。

  2. **init 0 **

shutdown命令
作用

​ 关机

格式

​ shutdown [选项] 时间 [消息]

选项 描述
-t seconds 设定在几秒钟之后进行关机程序
-k 并不会真的关机，只是将警告讯息传送给所有使用者
-r 关机后重新开机
-h 关机后停机
-n 不采用正常程序来关机，用强迫的方式杀掉所有执行中的程序后自行关机
-c 取消目前已经进行中的关机动作
time 设定关机的时间
message 传送给所有使用者的警告讯息
例子
  • 立即关机

# shutdown -h now

  • 指定10分钟后关机

# shutdown -h 10

  • 在关机动作触发前取消关机动作

# shutdown -c

  • 重新启动计算机

# shutdown -r now

init命令
作用

​ 切换运行模式

格式

​ init 运行级别代码

代码 描述
0 关机级别
1 单用户运行级别
2 多用户，字符模式，但系统不会启动NFS
3 多用户，字符模式，系统启动具有网络功能，常用运行级别
4 用户自定义级别
5 图形界面模式，常用运行级别
6 重启级别
halt命令
作用

​ 关机。先检测系统的runlevel，若runlevel为0或6，则关闭系统，否则即调用shutdown -h来关闭系统

格式

​ halt [选项]

选项 描述
-d 不在wtmp中记录（记录开关机等信息的二进制文件）
-f 不论目前的runlevel为何，不调用shutdown即强制关闭系统
-i 在halt之前，关闭全部的网络界面
-n halt前，不用先执行sync
-p halt之后，执行poweroff关闭电源
-w 取消目前已经进行中的关机动作
例子

关闭系统后关闭电源

# halt -p

关闭系统，但不留下日记记录

# halt -d

重启

  1. shutdown -r now
  2. init 6
  3. reboot
reboot命令
作用

​ 重新启动计算机。若系统的 runlevel 为 0 或 6 ，则重新开机，否则以 shutdown 指令（加上 -r 参数）来取代

格式

​ reboot [选项]

选项 描述
-d 不把记录写到 wtmp 档案里
-f 强迫重开机，不呼叫 shutdown 这个指令
-i 在重开机之前，关闭全部的网络界面
-n 在重开机前不做将记忆体资料写回硬盘的动作
-w 并不会真的重开机，只是把记录写到 /var/log/wtmp 档案里

Linux交互方式

Shell

Shell是一个作为用户与Linux系统间接口的程序，它允许用户向操作系统输入需要执行的命令，返回执行结果

种类

  1. ash：是贝尔实验室开发的Shell，它是轻量级的Shell，用于嵌入式设备或资源受限的系统中。bsh是对ash的符号链接。
  2. bash：是GNU的Bourne Again Shell，是Linux和其他Unix操作系统上最常用的Shell之一。它包括了Bourne Shell的所有功能，并且添加了许多扩展功能。在大多数Linux发行版中，bash是默认的Shell。sh以及bash2都是对它的符号链接。
  3. tcsh：是Berkeley UNIX C Shell，是C Shell的增强版本。它包括了C Shell的所有功能，并且添加了许多新特性，如命令补全、历史记录和别名等。csh是对它的符号链接。
  4. ksh：Korn Shell是由AT&T Bell实验室的David Korn开发的Shell。它的语法与Bourne Shell相同，同时具备了C Shell的易用特点。ksh在Unix系统中得到广泛使用，并且是POSIX标准的一部分。
  5. zsh：Z Shell是Korn Shell的一个增强版本，它具备bash Shell的许多特点。zsh包括了自动补全、历史记录、别名扩展、动态加载模块等许多功能，使得它成为一个强大的Shell。在许多Linux发行版中，zsh已经成为一个常见的替代品。

基本格式

命令名 [选项] <参数1> <参数2> ……

功能

Shell提示符

Shell提示符通常用于在命令行界面显示当前用户的用户名、主机名和工作目录等信息。在大多数Unix和Linux系统中，root用户的提示符是#，而其他用户的提示符是$。

命令自动补齐

命令自动补齐是一种Shell特性，可以在输入命令时按下Tab键，自动补齐命令名称、文件名和路径等信息，提高了命令输入的效率。

历史记录查看

历史记录是Shell保存用户执行的命令的列表，可以通过键盘上下键或者使用history命令查看之前执行的命令，并且可以重新执行之前执行的命令，提高了命令输入的效率。

  6. ls /usr /abc 1>/dev/null 2>&1：将ls /usr /abc命令的标准输出重定向到/dev/null设备文件中，将标准错误输出重定向到标准输出中，这样既不会输出标准输出，也不会输出错误信息。

在Linux系统中，数字1表示标准输出，数字2表示标准错误输出。/dev/null是一个特殊的设备文件，它可以将数据丢弃，相当于一个黑洞。因此，将输出重定向到/dev/null设备文件中，可以将输出数据丢弃。2>&1表示将标准错误输出重定向到标准输出，这样就可以将错误信息输出到与标准输出相同的位置，方便查看。

  • 默认情况下，这些特殊用户是无法登录的，如果给这些用户授权登陆口令后，就可以使用这些用户登录系统
  • /sbin/nologin 无法登陆shell
  • 举个例子来说，如果你安装了一个数据库服务，通常会创建一个专门的系统用户来运行这个服务，这个用户没有登录系统的权限，也没有终端Shell，只能执行与这个服务相关的操作。这个用户通常拥有较低的权限，只能读取、写入和修改数据库中的数据，不能执行其他的操作。这样做的好处是，即使攻击者通过某些漏洞获取了这个服务的权限，也无法通过这个用户的权限直接访问系统和其他的服务，从而保护了系统的安全性。

Linux文件存储

文件结构

  1. 名字
  2. 索引节点，又称I节点，存放文件的状态信息的一个数据结构。包含文件的长度、创建及修改时间、权限、所属关系、磁盘位置等。
  3. 磁盘数据

硬链接：文件（副本）

软链接：文件、目录（1、长路径->短路径，方便；2、解决磁盘空间不足问题[跨文件系统链接]）

区别：

  1. 文件类型不同：硬连接的连接文件和原文件在文件系统上没有区别，都是普通文件；软连接则是一个特殊的文件类型。
  2. 文件属性不同：硬链接和原文件的所有属性（如文件类型、权限、所有者等）都是相同的，而软链接则有自己的文件属性，它的文件类型是链接文件，而不是原文件的类型。
  3. inode不同：硬链接和软链接都是链接，但它们链接不同inode节点，因为硬链接和原文件共享同一个inode节点，而软链接则指向了一个新的inode节点。
  4. 链接数不同：硬链接和原文件共享同一个inode节点，它们的链接数相同，而软链接的链接数始终为1。
  5. 跨文件系统：硬链接不能跨越不同的文件系统，软链接可以跨越不同的文件系统。
  6. 修改方式不同：修改硬链接和软链接的方式不同，修改硬链接等同于修改原文件，而修改软链接只是修改链接文件，不会修改目标文件。
  7. 适用场景不同：硬链接只能连接文件，不能连接目录；软链接可以连接文件和目录。
  8. 对链接进行操作（修改、删除）对原文件的影响：修改任何一个硬链接文件，都会改变文件内容，删除任何一个硬链接文件并不影响其他相同的inode号的文件，直到删除所有硬链接和源文件,才能删除改文件；删除、修改软链接不会影响原文件。
  9. 对原文件进行操作（修改、删除）对链接的影响：硬链接同上；原文件被删除那么软链接失效，修改源文件也不会改变链接文件的内容。

联系：

  1. 硬连接和软连接都是指向原文件的指针，链接文件的存在依赖于原文件，删除原文件会使链接文件失效。
  2. 硬连接和软连接都是为了方便文件的共享和管理。

总的来说，硬链接更像是对原文件的副本，而软链接更像是对原文件的快捷方式。使用哪种方式取决于具体的需求和应用场景。

常见正则表达式

  • ^ 锚定行的开始 如：’^grep’匹配所有以grep开头的行
  • $ 锚定行的结束 如：’grep$’匹配所有以grep结尾的行
  • . 匹配一个非换行符的字符 如：’gr.p’匹配gr后接一个任意字符，然后是p
  • * 匹配零个或多个先前字符 如：’grep’匹配所有一个或多个空格后紧跟grep的行。.*一起用代表任意字符
  • [] 匹配一个指定范围内的字符，如’[Gg]rep’匹配Grep和grep
  • [^] 匹配一个不在指定范围内的字符，如：’[^A-FH-Z]rep’匹配不包含A-F和H-Z的一个字母开头，紧跟rep的行
  • \(..\) 标记匹配字符，如’\(love\)’，love被标记为1
  • \< 锚定单词的开始，如:’\<grep’匹配包含以grep开头的单词的行
  • \> 锚定单词的结束，如’grep\>’匹配包含以grep结尾的单词的行
  • x\{m\} 重复字符x，m次，如：’0\{5\}’匹配包含5个o的行
  • x\{m,\} 重复字符x,至少m次，如：’o\{5,\}’匹配至少有5个o的行
  • x\{m,n\} 重复字符x，至少m次，不多于n次，如：’o\{5,10\}’匹配5–10个o的行
  • \w 匹配文字和数字字符，也就是[A-Za-z0-9]，如：’G\w*p’匹配以G后跟零个或多个文字或数字字符，然后是p
  • \W \w的反置形式，匹配一个或多个非单词字符，如点号句号等
  • \b 单词锁定符，如: ‘\bgrep\b’只匹配grep

第七讲

网络模式

独立的守护进程工作模式（stand-alone）

  1. Unix传统的C/S模式的访问模式
  2. 在Client/Server模式下，服务器监听（Listen）在一个特定的端口上等待客户连接，连接成功后服务器和客户端通过端口进行数据通信
  3. 守护进程的工作就是打开一个端口，并且等待（Listen）进入连接
  4. 如果客户端发起一个连接请求，守护进程就创建（Fork）一个子进程响应这个连接，而主进程继续监听其他的服务请求
  5. 运行独立的守护进程工作方式称作：stand-alone

基于xinetd的工作模式

  1. 从守护进程的概念可以看出，对于系统所要提供的每一种服务，都必须运行一个监听某个端口连接情况的守护进程，这通常意味着资源浪费
  2. 为了解决这个问题，Linux引进了“网络守护进程服务程序”的概念
  3. RHEL5使用的网络守护进程是xinetd ( eXtended InterNET daemon)
  4. 和stand-alone模式相比xinetd模式也称 Internet Super-Server(超级服务器)
  5. xinetd能够同时监听多个指定的端口，在接受用户请求时，它能够根据用户请求的端口不同，启动不同的网络服务进程来处理这些用户请求
  6. 可以把xinetd看做一个管理启动服务的管理服务器，它决定把一个客户请求交给那个程序处理，然后启动相应的守护进程

运行单个xinetd就可以同时监听所有服务端口，这样就降低了系统开销，保护系统资源

但是对于访问量大、经常出现并发访问时，xinetd想要频繁启动对应的网络服务进程，反而会导致系统性能下降

特点
  • 支持对TCP、UDP、RPC服务的管理
  • 可以实施基于时间段的访问控制
  • 功能完备的log功能，可以记录连接成功、连接失败的行为
  • 能够有效地防止拒绝服务（DoS）的攻击
  • 能够限制同时运行的同一类型的服务器的数目
  • 能够限制log文件大小
  • 能够将某个服务绑定在特定的系统接口上，从而实现只能允许私有网络访问某项服务
  • 能够实现作为其它系统的代理

区别与联系

​ stand-alone模式

在stand-alone模式下，服务程序会作为一个独立的进程运行。这种模式下，服务程序需要监听特定的端口来接收来自客户端的请求，并直接与客户端通信，完成相应的操作。由于服务程序在独立的进程中运行，所以服务程序的配置、管理和调试都比较简单，但是这种模式下不能实现服务的并发处理。

在standalone模式下，服务程序作为一个独立的进程运行，它需要监听特定的端口来接收来自客户端的请求，并直接与客户端通信，完成相应的操作。在这种模式下，如果有多个客户端同时请求连接服务程序，则服务程序只能按照一个客户端一个客户端的方式来处理请求，无法同时处理多个请求。因此，standalone模式下不能实现服务的并发处理。

举个例子来说明，假设有一个基于TCP协议的文件传输服务，它可以实现将本地文件上传到服务器或从服务器下载文件到本地。在standalone模式下，服务程序会作为一个独立的进程运行，并监听特定的端口（例如端口号为8000）。当一个客户端请求连接服务程序时，服务程序会响应该请求并与该客户端进行通信，完成文件传输的操作。但是，如果同时有多个客户端请求连接服务程序，则服务程序只能按照一个客户端一个客户端的方式来处理请求，无法同时处理多个请求，从而导致服务的并发处理能力较弱。

因此，如果需要实现高并发的服务，就需要使用一些支持并发处理的模式，例如xinetd模式或者多线程模式。在这些模式下，服务程序可以同时处理多个客户端的请求，从而提高服务的并发处理能力。

​ 运用场景

Web服务器Apache和邮件服务器Sendmail、域名服务器Bind使用独立守护进程模式启动，因为这些负载很大服务器上，预先创建子服务器，可以提高客户的服务速度。

​ xinetd模式

在xinetd模式下，服务程序并不直接监听特定的端口，而是由xinetd守护进程来监听端口。当有客户端请求连接时，xinetd守护进程会启动一个服务进程来处理该请求。服务进程完成请求后，xinetd会将其关闭。这种模式下，服务程序可以实现并发处理，但是服务进程的启动和关闭会带来额外的系统开销。xinetd还可以控制服务进程的并发连接数和重启次数等参数。

联系：stand-alone模式和xinetd模式都是在Unix和Linux系统中运行网络服务的两种常见方式。它们都需要服务程序监听特定的端口，并能够响应客户端请求。不同的是，stand-alone模式下服务程序作为独立进程运行，而xinetd模式下服务程序由xinetd守护进程管理。在实际应用中，可以根据实际需求选择不同的模式来运行网络服务。例如，如果服务需要支持并发连接，则可以选择xinetd模式；如果服务需要简单、快速地运行，则可以选择stand-alone模式。

​ 运用场景

xinetd模式适用于需要同时支持多个服务，或需要支持并发连接的场景，例如SSH服务、Telnet服务、FTP服务等。在xinetd模式下，服务程序并不直接监听特定的端口，而是由xinetd守护进程来监听端口。当有客户端请求连接时，xinetd守护进程会启动一个服务进程来处理该请求。服务进程完成请求后，xinetd会将其关闭。这种模式下服务程序可以实现并发处理，但是服务进程的启动和关闭会带来额外的系统开销。

举个例子来说，vsftpd (Very Secure FTP Daemon)就是一个典型的xinetd模式下运行的FTP服务器。vsftpd服务器在监听21端口后，接收客户端的FTP请求并进行处理，支持多用户同时登录和并发文件传输等操作，因此需要使用xinetd模式来保证服务的性能和可靠性。

两种工作模式

PORT模式

主动模式PORT(服务器 主动连接 客户端 )

FTP服务器“主动”去连接客户端的数据端口来传输数据

  1. 客户端从一个任意的非特权端口N（N>1024）连接到FTP服务器的命令端口（即tcp 21端口）

  2. 客户端开始监听端口N+1，并发送FTP命令“port N+1”到FTP服务器

  3. 服务器会从它自己的数据端口（20）“主动”连接到客户端指定的数据端口（N+1）

  4. 客户端就可以和ftp服务器建立数据传输通道了

PASV模式

被动模式PASV(服务器 被动等待 客户端连接)

指的是FTP服务器“被动”等待客户端来连接自己的数据端口

  1. 客户端打开两个任意的非特权本地端口（N >1024和N+1）

  2. 第一个端口连接服务器的21端口，提交PASV命令

  3. 服务器会开启一个任意的非特权端口（P > 1024），并发送给客户端

    在PASV模式下，第三步服务器会发送给客户端一个包含了服务器IP地址和一个指定的非特权端口号的响应。这个非特权端口号就是服务器为数据连接所开放的端口号，客户端需要使用这个端口号来连接服务器的数据端口。

    例如，服务器可以发送如下形式的响应： 227 Entering Passive Mode (192,168,1,100,19,215)

    其中，括号中的前四个数字表示服务器的IP地址（192.168.1.100），后两个数字表示服务器为数据连接开放的端口号（4911 = 19*256 + 215）。

    客户端收到服务器的响应后，会使用指定的非特权端口号向FTP服务器发起数据连接，这个连接将会被用来传输数据。

  4. 客户端发起从本地端口N+1到服务器的端口的连接用来传送数据。（注意此模式下的FTP服务器不需要开启tcp 20端口了）

-w 禁止所有警告

VFS的功能和作用

Linux引进ext文件系统时有了一个重大的改进：为了更好的支持多种不同的文件系统，把文件系统从操作系统和系统服务中分离出来，在它们之间使用了一个接口层，也就是虚拟文件系统VFS(Virtual File System)

VFS功能包括：

  1. 记录可用文件系统的类型
  2. 将设备与对应的文件系统联系起来
  3. 处理面向文件的通用操作
  4. 涉及到针对文件系统的操作时，把他们映射到相关的物理文件系统

VFS 是 Linux 内核中的软件层，它在内核中提供了一组标准的、抽象的文件操作，允许不同的文件系统实现共存，并向用户空间程序提供统一的文件系统接口

通过VFS将不同文件系统的实现细节隐藏起来，从外部看上去，所有的文件系统都是一样的

进程

概念

Linux系统上所有运行的东西都可以称之为一个进程，如每个用户任务、每个系统管理任务

进程是一个程序的运行

进程与程序是有区别的

  1. 程序只是一个静态的指令集合，不占系统的运行资源，只占用磁盘空间
  2. 进程是一个随时都可能发生变化的、动态的、使用系统运行资源(cpu,内存等)的程序
  3. 一个程序可以启动多个进程

类型

Linux操作系统包括三种不同类型的进程，每种进程都有自己的特点和属性：

  • 交互进程: 由shell启动的进程

  • 批处理进程: 这种进程和终端没有联系，是一个进程序列

  • 守护进程: 在后台持续运行的进程

进程启动

前台启动
  • 用户键入一个命令，回车执行，就已经启动了一个前台的进程
  • 适合需要实时反馈的操作
  • 前台进程会一直占用终端，直到进程执行结束
  • ctrl+c 强制终端进程执行
  • kill 杀死进程
后台启动

​ 一般不需要与用户交互或者不需要实时的反馈

​ 在命令结尾上加上 &

常用操作

  • jobs：查看所有后台进程列表
  • ctrl+z：前台 -> 后台，并暂停执行
  • fg : 将后台进程调至前台执行
  • bg : 将一个后台暂停进程，继续执行

在 Linux 中，HUP 是 SIGHUP（Hang Up）信号的缩写。当进程接收到 SIGHUP 信号时，通常会触发重新加载配置或重启的操作。

关于为什么进程编号为 1 的进程（通常是 init 或 systemd 进程）会接收到 SIGHUP 信号，这与进程的特殊角色和启动过程有关。

在 Linux 启动过程中，init 或 systemd 进程是第一个被启动的进程，并作为所有其他进程的祖先进程（即进程树的根）。当终端连接断开或关闭时，会向进程组发送 SIGHUP 信号。因为 init 或 systemd 进程是进程组的领导者，所以它会收到该信号。

对于 init 或 systemd 进程，接收到 SIGHUP 信号通常会触发重新加载配置或重新启动系统服务的操作，以适应终端连接的变化。这样可以确保在断开或关闭终端连接时，系统能够正确地进行清理和重新配置。

需要注意的是，进程编号为 1 的进程并不总是 init 或 systemd，因为特定的系统配置和发行版可能会有所不同。但在大多数常见的 Linux 系统中，1 号进程是 init 或 systemd 进程。

在 Linux 中，kill 命令用于向进程发送信号。每个信号都有一个唯一的编号，用于指示要发送的特定操作。其中，信号编号 9 代表 SIGKILL，也称为强制终止信号。

SIGKILL 信号是一种不可被阻塞、不可被忽略的信号，用于强制终止目标进程。当使用 kill -9 <PID> 命令时，将发送 SIGKILL 信号给指定进程，使其立即终止，无法进行任何清理或处理操作。

为什么会选择使用 SIGKILL（信号编号 9）来强制终止进程呢？这是因为 SIGKILL 是一种终极手段，可以确保目标进程立即终止，无论其当前状态如何。这对于某些情况下需要迅速终止进程的场景非常有用，例如占用过多资源、无响应、无法正常退出等情况。

然而，需要注意的是，由于 SIGKILL 是无法被阻塞或忽略的信号，它会立即终止进程，可能会导致数据丢失或不可恢复的情况。因此，在正常情况下，最好首先尝试使用其他信号（如 SIGTERM，信号编号 15）来请求进程正常退出，以便进程有机会进行清理和释放资源。只有在必要时才使用 SIGKILL 信号来强制终止进程。

在 Linux 中，TERM 是 SIGTERM 信号的缩写。SIGTERM 是一种终止信号，用于请求目标进程正常退出。

当使用 kill 命令发送 SIGTERM 信号给目标进程时，进程会收到该信号并有机会进行清理、保存数据和释放资源等操作，然后正常退出。

为什么选择信号编号 15 来表示 SIGTERM 呢？这是由于早期 UNIX 系统的发展历史。在早期的 UNIX 系统中，信号编号 1（SIGHUP）和 15（SIGTERM）是两个特殊的信号，常用于控制进程的终止。SIGHUP（挂起信号）通常被用于指示进程重新读取配置文件或重启，而 SIGTERM 则用于请求进程正常退出。

虽然 SIGTERM 是一个终止信号，但它可以被进程捕获并执行一些特定的操作，例如完成未完成的任务、保存数据或清理资源。相比之下，SIGKILL（信号编号 9）是一个无法被阻塞或忽略的强制终止信号，进程无法捕获它或执行任何操作，会立即终止。

因此，为了优雅地终止进程并给予它进行清理的机会，通常会首先尝试发送 SIGTERM 信号（编号 15），只有在进程无响应或无法正常退出时，才会考虑使用 SIGKILL 信号（编号 9）来强制终止进程。

X Window的作用和组成

什么是X Window

一种以位图方式显示的软件窗口系统

诞生于1984，比Microsoft Windows要早

是一套独立于内核的软件

X Window的组成

X Window系统由三个基本元素组成

X 服务端
  • 控制输入及输出设备并维护相关资源的程序，它接收输入设备的信息，并将其传给X Client，而将X Client传来的信息输出到屏幕上(在屏幕上构造方块(窗口)，然后画出里面的元素)
  • 每一套显示设备只对应唯一的X Server
  • 由系统供应商提供，通常无法被用户修改
  • 只是一个普通的用户程序
X 客户端
  • 应用程序的核心部分，它与硬件无关，每个应用程序就是一个X Client
  • X Client可以是终端仿真器(Xterm)或图形界面程序，它不直接对显示器绘制或者操作图形，而是与X Server通信，由X Server控制显示
  • X Client无法直接影响视窗行为或显示效果，它们只能发送一个请求给X Server，由X Server来完成这些的请求
  • 多种多样的X Client程序向X Server发出请求，由X Server运算得出结果，再显示到指定的地方去
X 通信通道

  • X通信通道的主体是xlib（X函数库）

  • X Client调用xlib，利用相应的通信功能向X Server发出请求

  • X Server完成任务之后，同样调用xlib把结果显示指点的设备上去
