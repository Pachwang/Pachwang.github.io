伪共享

伪共享是多线程操作位于同一缓存行的不同变量时，由于缓存失效而引发性能下降的问题。缓存行是CPU缓存的最小单位，一般是64个字节，当CPU读取数据时，会把数据所在的缓存行一起读取。如果多个线程同时修改同一个缓存行中的不同变量，就会导致缓存行失效，需要重新从内存中读取，这样就降低了性能。
为了避免伪共享，有以下几种方法：

  • 在变量之间填充一些无用的变量，使得每个变量占据一个缓存行
  • 使用消除了伪共享结构的类，如java.util.concurrent.atomic.LongAdder
  • 使用@sun.misc.Contended注解（java8），让编译器自动填充空间

第四章

AtomicLong与LongAdder

前者是通过CAS提供非阻塞的原子性操作，但在高并发下大量线程会同时去竞争一个原子变量，由于同时只有一个线程会成功，这就造成大量线程竞争失败，会通过无限循环来不断进行自旋尝试CAS的操作，而这会白白浪费CPU资源，而LongAdder则是利用多个线程去竞争多个资源的思路来设计的。
LongAdder是Java 8新增的一个原子类，可以用于高并发下的计数器操作。相比于AtomicLong，LongAdder的性能更好，因为它将单一的变量内部拆分成多个变量，每个线程操作一个变量，最终将所有变量的值求和得到最终结果，从而减少了线程之间的竞争。
具体来说，LongAdder内部维护了一个Base变量和一个Cell数组，Base变量的初始值为0，Cell数组的初始长度为2。每个Cell都是一个原子的Long变量，表示分段计数器的值。当多个线程同时对LongAdder进行累加操作时，LongAdder会将累加任务分散到Cell数组中的不同Cell中进行处理1。每个线程对应一个Cell，只有在Cell的竞争激烈时才会扩容。
当多个线程获取LongAdder的值时，LongAdder会将Base和Cell数组中所有Cell的值相加，得到最终的结果。
总体来说，LongAdder通过将单一变量拆分成多个变量以及通过分段的方式减少了多线程的竞争，从而提高了高并发下的性能表现。

在这个例子中，我们创建了一个LongAdder对象，并使用10个线程对计数器进行累加操作。每个线程执行了10000次累加操作。通过调用LongAdder对象的increment()方法，我们可以避免在多线程情况下的竞争，从而提高程序的性能。
最后，我们通过调用LongAdder对象的sum()方法，获取计数器的总和。在这个例子中，计数器的预期总和为100 * 10000 = 1000000。

1 当多个线程同时对同一个LongAdder对象进行累加操作时，LongAdder内部实际上维护了一个Cell数组，每个Cell都包含一个long类型的值。这些Cell可以看作是多个独立的计数器，当线程对LongAdder对象进行累加时，LongAdder会根据线程的hashcode值将累加任务分散到不同的Cell中进行处理，以此实现了并发操作的效果。
例如，假设有三个线程Thread1、Thread2、Thread3同时对一个LongAdder对象进行累加操作，这个LongAdder内部维护了3个Cell，那么累加操作会被分散到这3个Cell中进行，例如Thread1的累加操作会被分散到Cell1中，Thread2的累加操作会被分散到Cell2中，Thread3的累加操作会被分散到Cell3中。最终，所有Cell中的值会被累加到一起，得到LongAdder对象最终的累加结果。这种方式可以避免多个线程对同一个计数器进行争用，从而提高了并发操作的性能。
例如，假设有10个线程同时对LongAdder进行递增，其中前5个线程递增的值为1，后5个线程递增的值为2。那么Cell数组中将会有5个Cell，前5个Cell的值为1，后5个Cell的值为2。最终，LongAdder将所有Cell的值相加，得到的结果为15。这样就保证了在高并发情况下LongAdder的准确性和可靠性。

第五章

CopyOnWriteArrayList

CopyOnWriteArrayList 是 Java 并发集合框架中的一种，它是一个线程安全的 List，它通过复制底层数组的方式来实现并发更新，因此读操作不需要加锁，写操作则需要加锁。但是，如果使用 CopyOnWriteArrayList 的 get(int index) 方法获取指定位置的元素时，可能会发生并发问题。
具体来说，假设线程 A 和线程 B 同时调用 CopyOnWriteArrayList 的 get(int index) 方法获取同一个索引位置的元素，那么在 CopyOnWriteArrayList 内部会进行一次数组的复制操作，这会导致线程 A 和线程 B 都获取到不同的数组引用，因此线程 A 可能获取到索引位置处的旧元素，而线程 B 可能获取到索引位置处的新元素。
例如，假设 CopyOnWriteArrayList 内部的数组为 [1, 2, 3, 4]，线程 A 和线程 B 同时调用 get(2) 方法，那么 CopyOnWriteArrayList 内部会创建一个新数组 [1, 2, 3, 4]，并返回数组下标为 2 的元素。由于新数组是在获取元素时才创建的，因此线程 A 和线程 B 可能都获取到不同的数组引用，例如线程 A 获取到的是旧数组 [1, 2, 3, 4]，而线程 B 获取到的是新数组 [1, 2, 5, 4]。因此，线程 A 可能获取到索引位置 2 上的旧元素 3，而线程 B 可能获取到索引位置 2 上的新元素 5。
要避免这种并发问题，可以在获取元素时加锁，或者使用类似 AtomicIntegerArray 的线程安全数组。另外，由于 CopyOnWriteArrayList 适用于读多写少的场景，因此在需要频繁修改的场景下，不建议使用 CopyOnWriteArrayList。

弱一致性迭代器

迭代器遍历的数组是一个快照
CopyOnWriteArrayList是一种线程安全的List集合，它的迭代器也是一种弱一致性迭代器。弱一致性迭代器的特点是在迭代过程中，如果有其他线程对集合中的元素进行了修改，那么当前迭代器的视图不一定能看到这些修改。
具体来说，当CopyOnWriteArrayList的迭代器被创建时，它会获取一份集合的快照，这个快照是迭代器可见的视图，迭代器在迭代时只会遍历这份快照中的元素。如果其他线程在迭代过程中修改了集合中的元素，这些修改不会影响到迭代器的遍历过程，因为迭代器的视图已经被固定了。

第六章

LockSupport

LockSupport 是 Java 提供的一个线程同步工具，用于实现类似于 wait/notify 的等待/通知机制。它是一种比 synchronized 更加灵活和高效的线程同步方式。
LockSupport 的主要作用是阻塞和唤醒线程，其底层是使用了操作系统提供的底层函数（如 Linux 的 futex）来实现线程阻塞和唤醒。
LockSupport 类提供了 park() 和 unpark() 两个方法。park() 方法用于阻塞当前线程，unpark() 方法用于唤醒指定线程。
park() 方法可以响应中断和超时，并且在被唤醒时不会出现“虚假唤醒”的情况（也就是说，即使没有调用 unpark() 方法，线程也不会在 park() 方法中无限期地阻塞）。
LockSupport 的使用一般是配合其它的同步工具一起使用，比如 ReentrantLock、Semaphore 等。

AQS

锁的底层支持

AQS底层基于一个双向队列（FIFO）和一个CAS原子变量，用于实现阻塞和唤醒线程的操作。当一个线程请求锁时，如果锁已经被占用，则当前线程会被阻塞，并被放入等待队列中。当持有锁的线程释放锁时，AQS会唤醒等待队列中的第一个线程，将其从等待队列中取出，并尝试将其设置为锁的持有者。
AQS中的状态变量表示锁的状态，通过CAS原子操作来改变状态。当一个线程成功获取到锁时，会将状态变量设置为占用状态，并将自己设置为锁的持有者。当释放锁时，会将状态变量设置为未占用状态，并唤醒等待队列中的第一个线程。

ConditionObject

ConditionObject是AQS的内部类，是用来支持锁的条件等待和通知的。ConditionObject的主要作用是提供了一个类似Object.wait()和Object.notify()的机制，可以让线程在等待某个条件成立时挂起，并在条件满足时被唤醒。
ConditionObject的实现是基于一个FIFO双向链表的，每个链表节点对应一个等待线程。当一个线程调用了Condition.await()方法后，它会被放到这个链表的末尾，然后等待被唤醒。当有其他线程调用了Condition.signal()或Condition.signalAll()方法时，会从这个链表的头部开始唤醒线程，直到链表为空或者条件被满足。
ConditionObject内部维护了一个整数类型的状态值state，当调用await()方法时，线程会释放锁并进入阻塞状态，同时会将这个状态值保存在一个等待队列中。当其他线程调用signal()或signalAll()方法时，会唤醒等待队列中的一个或多个线程，并将它们从等待队列中移除。唤醒的线程会重新尝试获取锁，并从await()方法中返回，继续执行后面的代码。
总的来说，ConditionObject为锁的使用者提供了一种灵活的等待和通知机制，可以更好地满足一些高级应用场景的需求。

条件变量支持

一个锁对应一个AQS阻塞队列，对应多个条件变量，每个条件变量有自己的一个条件队列。

AQS（AbstractQueuedSynchronizer）提供了对条件变量的支持，条件变量是用来对线程进行等待和唤醒的机制。AQS中通过内部类ConditionObject来实现条件变量的支持，每个条件变量对象都与一个独立的等待队列（WaitQueue）相关联，等待队列中存储了所有正在等待条件变量的线程节点（Node）。
ConditionObject类中定义了**await()、signal()和signalAll()等方法，用来实现线程等待和唤醒操作。当一个线程调用await()方法时，它会被加入到条件变量的等待队列中，然后被阻塞，直到被其他线程调用signal()或signalAll()**方法唤醒。
在AQS中，一个锁对象可以有多个条件变量对象。当一个线程持有一个锁对象时，它可以创建多个条件变量对象，并在条件变量对象上进行等待和唤醒操作。每个条件变量对象都是独立的，它们之间没有任何关联。

ReentrantLock

类图结构

ReentrantLock是可重入的独占锁，默认非公平锁，AQS的state状态值表示线程获取该锁的可重入次数。

获取锁与释放锁

  • 获取锁：当线程通过lock()方法获取ReentrantLock的锁时，首先会判断当前线程是否已经持有该锁，如果是，则直接增加该线程持有的锁计数器即可，否则会尝试使用CAS操作去获取锁。如果获取成功，则将当前线程与该锁建立关联，并将该线程持有的锁计数器置为1，同时返回。
  • 释放锁：当线程通过unlock()方法释放ReentrantLock的锁时，首先会判断当前线程是否持有该锁，如果不是，则抛出IllegalMonitorStateException异常，否则将该线程持有的锁计数器减1，如果减1后计数器的值为0，则该线程释放该锁。

公平锁与非公平锁

在公平锁模式下，线程尝试获取锁的顺序按照它们发出请求的顺序进行，也就是说，先到先得。在 ReentrantLock 中，公平锁模式是通过 FairSync 实现的。
在非公平锁模式下，线程在尝试获取锁时不考虑自己以前的等待时间，而是直接尝试获取锁。这种模式可能会比公平锁模式更快，因为它允许线程插队并在竞争到锁时获得它。在 ReentrantLock 中，非公平锁模式是通过 NonfairSync 实现的。
公平锁和非公平锁的代码区别在于获取锁的方法不同：

  • 公平锁：tryAcquire(int acquires) 方法会调用父类的tryAcquire方法，而父类的tryAcquire方法会判断同步队列是否有前驱节点，如果有则表示有线程正在等待获取锁，此时当前线程就会进入同步队列等待前驱节点释放锁。
  • 非公平锁：tryAcquire(int acquires) 方法会直接调用父类的nonfairTryAcquire方法，尝试通过CAS更新state状态值，如果更新成功则表示获取到了锁，否则进入同步队列等待唤醒。

ReentrantReadWriteLock

采用读写分离的策略，允许多个线程可以同时获取读锁，主要是为了满足写少读多的场景。

类图

读写锁内部维护了一个ReadLock和一个WriteLock,依赖Sync实现具体功能。state高16位表示读状态，低16位表示获取写锁的可重入次数。

写锁的获取与释放

独占锁，可重入。
如果当前没有线程获取到读锁和写锁，那么当前线程可以直接获取到写锁然后返回；如果当前已经有线程获取到读锁和写锁，那么请求写锁的线程会被挂起。

读锁的获取与释放

如果没有线程持有写锁，则当前线程可以获取写锁并将AQS的状态值state高16位增加1；如果有线程持有写锁，持有写锁的不是当前线程，那么当前线程会被阻塞挂起，持有写锁的是当前线程那么也可以获取读锁。释放锁，循环判断AQS的状态值，如果为0，则说明没有读线程占用读锁，释放一个由于获取写锁被阻塞的线程；如果不为0，还有其他线程持有读锁。

第七章

阻塞队列使用锁来实现，非阻塞队列使用CAS非阻塞算法实现

ConcurrentLinkedQueue

线程安全的无界无阻塞队列，底层是单向链表。结点类型为Node，内部维护了一个volatile修饰的变量item，用来存放结点的值，next存放链表的下一结点。

offer 操作

在队列末尾添加一个元素
通过cas操作来控制某时只有一个线程可以追加元素到队列末尾。竞争失败的线程会通过循环一次次尝试进行cas操作，直到成功才返回，即通过使用无限循环不断进行cas尝试方式来替代阻塞算法挂起调用线程。

add 操作

在内部调用的还是offer方法

poll 操作

在队列头部获取并移除一个元素，只是通过使用cas将当前结点的item值设置为null，然后重新设置头结点，被移除的结点在垃圾回收时被回收掉。

peek 操作

不同于poll，只是获取而不移除队列头部的一个元素，第一次调用时会删除哨兵结点，也就是 next 为 null 的结点，此后head才会指向第一个真正的队列元素。

remove 操作

删除元素，存在即删除，存在多个则删除第一个。

size 操作

计算当前队列元素个数，存在线程安全问题，cas没有加锁，计数不准确。

contains 操作

判断队列中是否含有指定对象，存在线程安全问题。

LinkedBlockingQueue

亦是用单向链表实现的，结点类型为Node，但是有一个 初始值为0的原子变量count，用来记录队列元素个数；两个ReentrantLock实例 takeLock和putLock，分别用来作为独占锁控制元素出队和入队的原子性操作；两个条件变量notEmpty、notFull，它们内部有一个条件队列分别用来存放出队和入队时阻塞的线程。

当某线程要插入新元素但是队列已满，那么阻塞并将其放入notFull内部维护的条件队列中进行等待，直到有线程取出了元素并且调用了notFull的signal方法（表示现在已经不是满的）；
当某线程需要从队列取出元素但是队列是空的，那么阻塞并将其放入notEmpty内部维护的条件队列中进行等待，直到有线程插入了元素并且调用了notEmpty的signal方法（表示现在已经不是空的了）。

独占锁创建过程

takeLock

执行take、poll等操作时需要获取takeLock，获取到锁，保证同时只有一个线程能操作链表头结点，当队列为空，执行出队操作的线程会被放入到notEmpty条件队列中。

putLock

执行put、offer等操作时需要获取takeLock，获取到锁，保证同时只有一个线程能操作链表尾结点，当队列为满，执行入队操作的线程会被放入到notFull条件队列中。

offer 操作

入队，在队列尾部插入一个元素，该方法是非阻塞的，插入成功返回true，插入失败则丢弃此元素。

put 操作

与offer操作类似但却是阻塞的，也是在队列尾部插入一个元素，如果队列有空闲则插入后直接返回，如果队列已满则阻塞当前线程直到有空闲才插入成功后返回。

poll 操作

出队，从队列头部获取并移除一个元素，该方法是非阻塞的。

take 操作

出队，从队列头部获取并移除一个元素，该方法是阻塞的，队列为空则阻塞直至队列不空然后返回元素。

peek 操作

从队列头部获取一个元素，该方法是非阻塞的。

remove 操作

offer(E e)
  1. 首先，方法会检查元素 e 是否为 null，如果是则抛出 NullPointerException 异常。
  2. 然后，方法会获取队列的锁，以确保在操作过程中不会被其他线程干扰。
  3. 接着，方法会定义变量 n、cap、es，分别表示队列的大小、容量和元素数组。然后使用 while 循环来检查队列是否已满，如果已满则调用 tryGrow(es, cap) 方法进行扩容，直到队列容量足够插入元素为止。

  5. 插入元素后，需要更新队列的大小，并使用 notEmpty.signal() 通知等待的 take() 操作可以获取元素了。
  6. 最后，释放锁，并返回 true 表示插入成功。
tryGrow(Object[] array, int oldCap)

tryGrow() 是 PriorityBlockingQueue 的一个私有方法，用于扩容队列的数组。这个方法会在队列已满时自动调用，以确保队列有足够的空间可以插入元素。
下面是方法的具体实现代码：

  1. 首先，方法会释放队列的锁，以允许其他线程在扩容期间访问队列。
  2. 然后，方法会尝试获取 allocationSpinLock，这是一个自旋锁，用于控制在同一时间只有一个线程可以进行扩容操作。如果当前没有其他线程正在进行扩容，则当前线程可以获取到锁，并开始扩容操作。
  3. 扩容操作会计算新的容量 newCap，根据当前队列容量的大小进行扩容。如果队列较小，则新容量会增加 2 个元素，以加快队列的增长速度；如果队列较大，则新容量会增加原来容量的一半。然后检查新容量是否超过了 MAX_ARRAY_SIZE 的限制，如果超过了，则抛出 OutOfMemoryError 异常。如果新容量合法，则创建一个新数组 newArray，其长度为 newCap。
  4. 扩容操作完成后，释放 allocationSpinLock，以允许其他线程进行扩容操作。
  5. 如果当前线程成功创建了新数组，则需要将原数组中的元素复制到新数组中。由于在释放锁后可能有其他线程修改了队列的状态，因此需要在获取锁之前进行 Thread.yield()，以等待其他线程完成扩容操作。
  6. 最后，重新获取锁，并将新数组设置为队列的底层数据结构，同时将原数组中的元素复制到新数组中。

整个扩容操作需要注意多线程并发访问的问题，因此需要采取一系列同步措施，以确保扩容操作能够顺利进行。这些措施包括使用自旋锁进行同步、释放锁以允许其他线程访问队列、在获取锁之前进行 Thread.yield() 以等待其他线程完成扩容操作等。同时，也需要进行一些安全性检查，例如检查新容量是否合法，以防止数组溢出或者内存泄漏等问题。

siftUpComparable(int k, T x, Object[] es)

siftUpComparable 方法是 PriorityBlockingQueue 类中的一个私有方法，用于将新的元素 x 插入到底层数组 es 中的指定位置 k，并在插入后保持堆的有序性。
此方法实现了对 Comparable 接口的支持，即如果元素类型 T 实现了 Comparable 接口，则优先使用其自然排序的方式来保持堆的有序性。
具体实现思路为：

  1. 首先将待插入的元素 x 强制转换为 Comparable<? super T> 类型，以方便进行比较。
  2. 从位置 k 开始向上逐层比较和交换元素，直到 x 找到了它在堆中的正确位置。
  3. 比较的方式是将 x 和其父节点的元素进行比较，如果 x 比父节点小，则将父节点下移，同时更新 k 的值为当前父节点的位置。
  4. 如果 x 比父节点大，则说明 x 已经在正确的位置上，插入操作结束。

最后，将 x 插入到最终的位置 k 上，完成插入操作。

dequeue()

dequeue 方法是 PriorityBlockingQueue 类中的一个私有方法，用于从底层数组中删除并返回队首元素，并在删除后保持堆的有序性。
具体实现思路为：

  1. 首先从底层数组 es 中取出队首元素，即位置为 0 的元素，并保存到 result 中。
  2. 如果队首元素不为空，则需要进行删除操作：
    1. 首先从底层数组中取出最后一个元素 x，并将其保存到 es 的最后一个位置上。
    2. 然后将队列大小 size 减一，并将队尾位置 n 更新为新的 size。
    3. 如果队列不为空，则需要将新的队首元素 x 向下逐层比较和交换元素，以保持堆的有序性。

    5. 比较的起点是位置 0，每次从两个子节点中选择较小的一个进行比较和交换，直到 x 找到了它在堆中的正确位置。
  3. 最后将队首元素 result 返回，完成删除操作。

需要注意的是，该方法在执行时必须要求持有锁，因此没有在方法签名中标注 synchronized 或 ReentrantLock 等关键字。

siftDownComparable(int k, T x, Object[] es, int n)

siftDownComparable 方法用于在 PriorityBlockingQueue 中下滤元素以满足堆的性质。这个方法的目的是将元素 x（在 PriorityBlockingQueue 中位于索引 k）下移，直到它在堆中的位置正确为止。
该方法使用了以下参数：

  • k：元素 x 在 PriorityBlockingQueue 中的索引。
  • x：需要下移的元素。
  • es：元素数组。
  • n：PriorityBlockingQueue 中元素的数量。

方法首先将元素 x 转换为可比较的类型，然后执行下述循环：

  1. 定义一个变量 half 表示非叶子节点的数量，其中 n >>> 1 表示右移一位，等价于除以 2。这里的 half 是一个索引值，表示最后一个非叶子节点的索引。
  2. 当 k 小于 half 时，进入循环。因为在 PriorityBlockingQueue 中，所有的叶子节点都出现在最后一层或者倒数第二层，所以只有索引小于 half 的节点才有可能是非叶子节点。
  3. 假设当前节点的左孩子为最小节点，将其索引赋值给变量 child。
  4. 如果当前节点存在右孩子，检查右孩子是否比左孩子小，如果是，将右孩子的索引赋值给变量 child。
  5. 如果 key 比 child 大或相等，则跳出循环。
  6. 否则，将 child 的值赋给 es[k]，更新 k 和 child 的值。
  7. 返回 es[k]。

在循环结束时，元素 x 将位于堆中正确的位置，即节点 k。最后将 x 放置在节点 k 处，并返回 es[k]，即原先位于节点 k 的元素，以便维护堆的有序性质。

Leader-Follower模式是一种常用的并发模式，也被称为”Reactor模式”。它是基于事件驱动模型的一种并发编程模式，主要用于构建高并发的网络应用程序。
在该模式中，系统中有两个主要的角色：Leader和Follower。Leader负责管理整个系统的状态和处理核心业务逻辑，Follower则负责处理请求和响应。
具体来说，当有新的请求到达系统时，Follower会将请求放入一个请求队列中，并通过某种方式通知Leader有新的请求需要处理。Leader则从请求队列中获取请求并进行处理，处理完成后将响应返回给对应的Follower进行发送。
这种Leader-Follower的架构可以有效地解决高并发网络应用程序中的性能瓶颈问题。通过将请求处理和响应发送分离，可以避免阻塞或延迟等问题的发生。同时，通过多个Follower的协作，可以进一步提高系统的吞吐量和响应能力。
不过，需要注意的是，在Leader-Follower模式中，Leader作为整个系统的中心节点，可能成为单点故障，因此需要考虑如何保证Leader的高可用性和容错性。此外，Leader和Follower之间的通信也需要进行一定的优化和协调，以避免性能瓶颈的发生。

线程池解决的问题

  1. 执行大量异步任务时线程池能够提供较好的性能。因为创建、销毁线程池是需要开销的；线程池里面的线程可以复用。

ThreadPoolExecutor 类概述

Integer类型的原子变量ctl，用来记录线程池状态和线程池中线程的个数。高3位用来表示线程池状态，低29位用来记录线程池中线程个数。

线程池状态

  • RUNNING：接收新任务并且处理阻塞队列中的任务
  • SHUTDOWN：拒绝新任务但是处理阻塞队列里的任务。
  • STOP：拒绝新任务并且抛弃阻塞队列里的任务，同时会中断正在处理的任务。
  • TIDYING：所有任务都执行完（包含阻塞队列中的任务）后当前线程池活动线程数为0。
  • TERMINATED：终止状态。

线程池参数

  • corePoolSize：线程池的核心线程数，即保持活跃的线程数，即使它们闲置也不会被回收。
  • maximumPoolSize：线程池的最大线程数，包括核心线程和临时线程。当任务队列满了之后，临时线程会被创建来处理任务，直到达到最大线程数。
  • keepAliveTime：当线程池中的线程数大于核心线程数时，临时线程的最大存活时间。如果线程在这段时间内没有执行任何任务，它将被终止。
  • unit：keepAliveTime的时间单位。
  • workQueue：任务队列，用于存储待执行的任务。线程池中的线程将从该队列中取出任务进行执行。
  • threadFactory：用于创建新线程的工厂。
  • handler：当任务队列和线程池都已满时，用于处理新任务的拒绝策略。

    3. keepAliveTime: 线程池中的线程空闲时间，即当线程池中的线程数量大于corePoolSize时，多余的空闲线程在被回收前等待新任务的最长时间。

    5. workQueue: 任务队列，用于保存等待执行的任务。在这个方法中，任务队列采用一个没有容量的同步队列SynchronousQueue实现，当有新任务到来时，必须要有空闲的线程去执行，否则新任务就会被拒绝。

  2. 如果当前线程数小于 corePoolSize，就会尝试使用 addWorker 方法添加一个新的工作线程来执行任务。addWorker 方法会自旋地尝试增加工作线程数量，直到增加成功或者失败。

  3. 如果队列可以接受新任务（队列大小未满），就会将任务加入到队列中。

  4. 二次检查，如果此时线程池的状态已经发生了改变（例如，线程池已经关闭或停止），则可能需要将任务从队列中移除并拒绝执行；如果当前线程池为空那么添加一个线程。

  5. 如果队列已满，就会尝试增加线程，最多增加到 maximumPoolSize。如果增加线程数失败，则说明线程池已经饱和，此时任务也无法执行，需要将其拒绝。

addWorker(Runnable firstTask, boolean core)

ThreadPoolExecutor 中的一个私有方法，用于向线程池中添加工作线程（Worker）。方法的主要作用是先检查线程池的状态和当前线程数，然后通过一个无限循环尝试添加工作线程，直到成功添加为止。

下面是具体的步骤：

  1. 首先，通过 ctl.get() 获取当前线程池的控制状态。这个状态是一个 int 类型的值，其中包括线程池的运行状态和当前的工作线程数等信息。
  2. 如果当前状态为 STOP 、TIDYING 或者 TERMINATED；当前线程池的状态为 SHUTDOWN 且任务队列为空，当前线程池的状态为 SHUTDOWN 且 firstTask 不为 null，则直接返回 false，表示无法添加工作线程。
  3. 然后进入一个无限循环，用于尝试添加工作线程。在循环中，先判断当前线程数是否已经达到了最大值。如果是，直接返回 false，表示无法添加工作线程。
  4. 如果当前线程数还没有达到最大值，则通过 compareAndIncrementWorkerCount(c) 尝试增加线程数。如果增加成功，就跳出循环，表示成功添加了一个工作线程；如果增加失败，则重新获取当前状态 c 并继续循环，如果线程池状态变化了则跳到外层循环，否则内层循环继续cas。
  5. 如果成功添加了一个工作线程，则需要将其加入到线程池中。首先创建一个 Worker 对象，然后将其中的线程对象 t 取出来，再通过 mainLock.lock() 获取线程池的锁，确保线程池状态不会被其他线程修改，因为可能有多个线程调用了线程池的execute方法。
  6. 在获取锁后，先重新获取当前的状态 c，然后判断线程池是否仍在运行或者处于 SHUTDOWN 状态但仍有任务需要执行。如果是，则将 Worker 对象添加到线程池的 workers 集合中，并将 workerAdded 标志设置为 true。同时，如果 workers 集合的大小超过了 largestPoolSize，则更新 largestPoolSize 的值。
  7. 通过 workerAdded 标志判断新增线程是否成功。如果需要启动，则通过 t.start() 启动线程，并将 workerStarted 标志设置为 true。
  8. 最后，如果启动线程失败，则调用 addWorkerFailed(w) 方法处理异常情况。如果线程启动成功，则返回 true，表示成功添加了一个工作线程。

第十章

CountDownLatch

应用场景

主线程中开启多个线程去并行执行任务，并且主线程需要等待所以子线程执行完毕后再进行汇总。此前一般都使用线程的join()方法来实现这一点，但join()方法不够灵活，不能满足不同场景的需要。

  1. 只能等待一个线程：join()方法只能等待一个指定的线程执行完毕后再继续执行，不能同时等待多个线程。
  2. 无法控制等待时间：调用join()方法时，主线程会一直等待被调用线程执行完毕，不能设置等待的超时时间。如果被调用线程发生阻塞或死锁，那么主线程就会一直等待，无法继续执行。
  3. 不够灵活：join()方法的调用位置和方式比较固定，无法灵活地控制多个线程的执行顺序和时间，也不能动态地控制等待的线程数量。
  4. 使用线程池来管理线程时一般都是直接添加Runable到线程池，这时候就没有办法再调用线程的join方法了。

原理探究

CountDownLatch是使用AQS实现的，实际上是把计数器的值赋给了AQS的状态变量state。

void await()方法

当前线程阻塞，直到所有线程都调用了CountDownLatch对象的countDown方法后，也就是计数器的值为0才会返回，或者其他线程调用了当前线程的interrupt()方法中断了当前线程然后返回。

boolean await(long timeout, TimeUnit unit)方法

线程阻塞，直到计数器值为0，或者设置的timeout到了，因超时返回false，或者其他线程中断当前线程。

void countDown()方法

线程调用后，计数器的值递减，如果计数器的值为0，那么唤醒所有因调用await方法而被阻塞的线程。

long getCount()方法

获取当前计数器的值，也就是AQS的state的值

CyclicBarrier

非常适合分段任务有序执行的场景，并使用条件变量来实现线程同步

概述

  1. CountDownLatch等到计数器值变为0后，再调用await或者countdown方法都会立刻返回，是一次性的，不能达到线程同步的效果。
  2. CyclicBarrier可以重置计数器。回环：当所有等待线程执行完毕，会重置CyclicBarrier的状态后它可以被重用。
  3. 屏障：线程调用await方法后就会被阻塞，这个阻塞的就是屏障点，等所有线程调用了await后，线程才会继续执行下去。
  4. 回环屏障，可以让一组线程全部达到一个状态后再全部同时执行。

多个线程之间是相互等待的，假如计数器值为N，那么随后调用await方法的N-1个线程都会因为到达屏障点而被阻塞，当第N个线程调用await后，计数器值为0了，这时候第N个线程才会发出通知唤醒前面的N-1个线程。

原理探究

  1. lock基于独占锁实现的，本质底层还是基于AQS的
  2. trip，独占锁的条件变量
  3. parties用来记录线程个数，含义是多少个线程调用了await后所有线程才会突破屏障继续往下执行
  4. count一开始等于parties，每当有线程调用await方法就递减1，当count减为0就表示所有线程到达了屏障点
  5. barrierCommand，当所有线程到达屏障点后执行的任务
  6. generation内部有一个变量broken，用来记录屏障是否被打破
  7. 每次count变为0后，会将parties的值赋给count
int await()方法
  • 所有线程到达屏障点然后返回
  • 被中断，抛出异常返回
  • 当前屏障点关联的Generation对象的broken标志设为true抛出异常返回
boolean await(long timeout, TimeUnit unit)方法

出了上面说的情况以外，设置的超时时间到了后返回

int dowait(boolean timed, long nanos)方法

await()的内部实现，当一个线程调用后，首先会得到独占锁，其余线程阻塞。获取到锁的线程执行完毕后被放入条件变量阻塞队列，并唤醒其他线程，从而其余线程竞争锁。当所有线程都被放入阻塞队列后，重置CyclicBarrier，这样的话他们又可以继续执行下去了

Semaphore

概述

同步器，内部计数器是递增的，可以初始化为一个特定的值而不需要知道待同步的线程个数，而是再需要同步的时候调用acquire方法时指定需要同步的线程个数

如果构造Semaphore时传递的参数为N，并在M个线程种调用了该信号量的release方法，那么在调用acquire方法使M个线程同步时传递的参数应该是M+N

原理探究

Semaphore内部还是使用AQS实现的。默认采用非公平锁。

void acquire()方法

阻塞，当前线程调用该方法的目的是希望获取一个信号量资源。如果当前信号量个数大于0，则当前信号量的计数会减1，然后返回；否则如果当前信号量个数为0，那么当前线程会被放入AQS的阻塞队列。

会对中断进行响应。如果当前线程被打断了，则会抛出中断异常。

void acquire(int permits)方法

获取permits个信号量

void release()方法

把当前Semaphore对象的信号量值增加1，如果当前有线程因为调用acquire方法被阻塞而被放入了AQS的阻塞队列，则会根据公平策略选择一个信号量个数能被满足的线程进行激活，激活的线程尝试获取信号量。

void release(int permits)方法

每次调用信号量值增加permits

ArrayBlockingQueue的使用

异步日志打印模型

让业务线程把要打印的日志任务放入一个队列后直接返回，然后使用一个线程专门负责从队列中获取日志任务并将其写入磁盘。这样的话，业务线程打印日志的耗时就仅仅是把日志任务放入队列的耗时了。

logback的异步日志模型是一个多生产者-但消费者模型，通过使用队列把同步日志打印转换成了异步，业务线程只需要通过调用异步appender把日志任务放入日志队列，而日志线程则负责使用同步的appender进行具体的日志打印，日志打印线程只需要负责生成日志并将其放入队列，不需要关系消费者线程何时把日志具体写入磁盘。

异步日志与具体实现

AsyncAppender是实现异步日志的关键，继承自AsyncAppenderBase，重写了其中的一些方法。

logback中异步日志队列是一个阻塞队列，其实就是有界阻塞队列ArrayBlockingQueue，其中

  • queueSize表示有界队列的元素个数，默认256个
  • worker是个线程，是异步打印日志中的单消费者线程
  • aai是一个appender的装饰器，里面存放同步日志的appender
  • appenderCount记录aai里面附加的同步appender的个数
  • neverBlock用来指示当日志队列满时是否阻塞打印日志的线程
  • discardingThreshold是一个阈值，当日志队列里面的空闲元素个数小于该值时，新的某些级别的日志会直接被丢弃

SimpleDateFormat是线程不安全的，多线程共用一个SimpleDateFormat实例对日期进行解析或者格式化会导致程序出错。

问题分析

每个SimpleDateFormat实例里面都会有一个Calendar对象，SimpleDateFormat之所以是线程不安全的，就是因为Calendar是线程不安全的，而Calendar之所以是线程不安全的，是因为其中存放日期数据的变量都是线程不安全的，比如fields、time。

解决方案

  1. 每次使用时都new一个SimpleDateFormat的实例，但是这样开销很大。
  2. 使用Synchronized进行同步，但加锁意味着多个线程竞争锁，在高并发场景下这会导致系统响应性能下降。
  3. 使用ThreadLocal，大大节省了对象的创建和销毁的开销并且不需要多个线程同步。

使用ThreadLocal不当可能导致内存泄漏

为什么会内存泄漏

ThreadLocal只是一个工具类，具体存放变量的是线程的threadLocals变量。threadLocals是一个ThreadLocalMap类型的变量。

ThreadLocalMap内部是一个继承自WeakReference的Entry数组，Entry内部的value用来存放通过ThreadLocal的set方法传递的值。

当一个线程调用ThreadLocal的set方法设置变量时，当前线程的ThreadLocalMap里就会存放一个记录，这个记录的key为ThreadLocal的弱引用，value则为设置的值。如果一直没有调用对应的remove方法，并且在其他地方还有对ThreadLocal的引用，那么当前线程的ThreadLocalMap变量里会存在对ThreadLocal变量的引用和对value对象的引用，它们是不会被释放的，这就造成了内存泄漏。

尽管ThreadLocalMap的Entry中的key使用的是对ThreadLocal对象的弱引用，但仍然可能存在key为null但是value不为null的entry项

内存泄漏

内存泄漏（Memory Leak）指的是程序在运行过程中由于某些原因，未能释放已经不再使用的内存空间，导致系统的可用内存逐渐减少，最终导致系统运行缓慢、不稳定甚至崩溃的问题。
在Java中，内存泄漏通常指的是程序中存在对对象的引用，但这些对象已经不再需要使用，却未能被垃圾回收器回收，从而导致内存空间被占用，最终导致系统内存不足或者系统运行缓慢的问题。
内存泄漏是一个非常严重的问题，如果不及时处理，可能会导致系统的可用内存越来越少，最终导致系统运行缓慢、不稳定甚至崩溃。因此，在编写程序时，应该特别注意避免内存泄漏问题的出现。在Java中，避免内存泄漏问题的一个重要方法是确保程序正确地使用对象引用和垃圾回收机制。
