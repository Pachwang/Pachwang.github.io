  2. String 、 StringBuffer 和 StringBuilder 的区别是什么? String 为什么是不可变的?

    三者的区别：

    String：String 对象是不可变的，因此也是线程安全的，每次对 String 类型进行改变的时候，都会生成一个新的 String 对象，然后将指针指向新的 String 对象。

    StringBuffer：StringBuffer 对象是可变的，它对方法加了同步锁或者对调用的方法加了同步锁，所以是线程安全的，StringBuffer 每次都会对对象本身进行操作。

    StringBuilder：StringBuilder 对象是可变的，它没有对方法进行加同步锁，所以是非线程安全的，StringBuilder 也是对对象本身进行操作的。

    为什么是不可变的：

    从源码层面：

    • 保存字符串的数组被 final 修饰且为私有的，并且 String 类没有提供/暴露修改这个字符串的方法。

    • String 类被 final 修饰导致其不能被继承，进而避免了子类破坏 String 不可变。

    从设计理念：

    • 安全性：由于String是不可变的，它在多线程环境下不需要额外的同步措施，从而避免了一些并发问题。
    • 缓存优化：因为字符串不可变，相同的字符串在多处引用时可以共享同一份数据，从而在一些情况下减少内存占用。

  3. String s1 = new String(“abc”);这段代码创建了几个字符串对象？

    • 如果字符串常量池中不存在字符串对象“abc”的引用，那么它将首先在字符串常量池中创建，然后在堆空间中创建，因此将创建总共 2 个字符串对象。
    • 如果字符串常量池中已存在字符串对象“abc”的引用，则只会在堆中创建 1 个字符串对象“abc”。

  4. == 与 equals的区别?hashCode 与 equals ?

  5. 包装类型的缓存机制了解么？

    Java 基本数据类型的包装类型的大部分都用到了缓存机制来提升性能。

    Byte,Short,Integer,Long 这 4 种包装类默认创建了数值 [-128，127] 的相应类型的缓存数据，Character 创建了数值在 [0,127] 范围的缓存数据，Boolean 直接返回 True or False。

  6. 自动装箱与拆箱了解吗？原理是什么？

    • 装箱：将基本类型用它们对应的引用类型包装起来；
    • 拆箱：将包装类型转换为基本数据类型；
    • 原理：装箱其实就是调用了 包装类的valueOf()方法，拆箱其实就是调用了 xxxValue()方法。

  7. 深拷贝和浅拷贝区别了解吗？什么是引用拷贝？

    • 浅拷贝：浅拷贝会在堆上创建一个新的对象（区别于引用拷贝的一点），不过，如果原对象内部的属性是引用类型的话，浅拷贝会直接复制内部对象的引用地址，也就是说拷贝对象和原对象共用同一个内部对象。
    • 深拷贝：深拷贝会完全复制整个对象，包括这个对象所包含的内部对象。
    • 引用拷贝：引用拷贝就是两个不同的引用指向同一个对象。

  8. 谈谈对 Java 注解的理解，解决了什么问题？

    • 可以看作是一种特殊的注释，主要用于修饰类、方法或者变量，提供某些信息供程序在编译或者运行时使用。
    • :robot: Java 注解是一种元数据（metadata）机制，它允许开发者在源代码中添加附加信息，用于提供关于程序元素（类、方法、字段等）的更多信息。注解本身不会直接影响程序的运行，而是为编译器、工具和框架提供了额外的信息，以便它们可以执行特定的操作或提供特定的功能。
    • 解决的问题：
      • 提供元数据信息：注解允许开发者向源代码中添加元数据信息，这些信息可以描述类、方法、字段等的用途、约束、配置等。这样可以为代码添加更多的上下文和含义，有助于代码的可读性和维护性。
      • 减少重复性代码：注解可以用于标记一些通用的模式，避免了在多个地方重复编写相同的代码。例如，JUnit框架使用注解来标记测试方法，从而减少了编写测试代码时的模板代码。
      • 编译时检查：一些注解可以用于在编译时进行静态检查，从而帮助捕获一些潜在的错误。例如，@Override 注解用于标记方法覆盖父类的方法，编译器会检查是否真的存在被覆盖的方法。
      • 配置和代码生成：很多框架和工具使用注解来进行配置和代码生成。例如，Spring框架使用注解来指定依赖注入的方式，以及标记事务边界等。
      • 文档生成：注解可以用于生成文档或API文档，通过提供更多的元数据信息，帮助生成更详细和准确的文档。

  9. Exception 和 Error 有什么区别？

    在 Java 中，所有的异常都有一个共同的祖先 java.lang 包中的 Throwable 类。Throwable 类有两个重要的子类:

    • Exception :程序本身可以处理的异常，可以通过 catch 来进行捕获。Exception 又可以分为 Checked Exception (受检查异常，必须处理) 和 Unchecked Exception (不受检查异常，可以不处理)。
    • Error：Error 属于程序无法处理的错误 ，不建议通过 catch 捕获 。例如 Java 虚拟机运行错误（Virtual MachineError）、虚拟机内存不够错误(OutOfMemoryError)、类定义错误（NoClassDefFoundError）等 。这些异常发生时，Java 虚拟机（JVM）一般会选择线程终止。

  10. Java 反射？反射有什么缺点？你是怎么理解反射的（为什么框架需要反射）？

    • 反射赋予了我们在运行时分析类以及执行类中方法的能力。通过反射你可以获取任意一个类的所有属性和方法，你还可以调用这些方法和属性。

    • 缺点：

      • 性能开销：由于反射需要在运行时进行动态查找和调用，相比直接调用方法和访问字段，其性能较差。频繁使用反射可能会导致程序性能下降。

      • 编译时类型检查：反射的使用在一定程度上减少了编译器的类型检查能力，可能会导致在运行时出现类型相关的错误。

      • 安全性问题：反射可以绕过访问控制符限制，访问私有方法和字段。这可能会导致一些安全性问题，特别是在开发不稳定或不可信的代码时。

    • :robot: 我理解反射是一种强大但需要谨慎使用的机制。它为程序员提供了在运行时检查和操作类、方法和字段等的能力，使得代码可以更加动态和灵活。然而，反射的性能和安全性问题需要仔细考虑，以避免出现潜在的性能瓶颈和安全漏洞。

      框架通常需要使用反射的主要原因是为了实现通用性和灵活性。框架需要能够处理不同的类和配置，而不需要在代码中硬编码这些信息。通过反射，框架可以在运行时获取类的信息、调用方法、访问字段等，从而使得框架可以适应不同的情况和用户需求。例如，依赖注入框架（如Spring）使用反射来实现将依赖自动注入到对象中，而不需要用户手动实例化对象和设置依赖。尽管反射的性能开销需要考虑，但在需要灵活性和通用性的框架中，反射是一个非常有价值的工具。

  11. Java 泛型了解么？什么是类型擦除？介绍一下常用的通配符？

    • 泛型 是一种在编译时提供类型安全性的机制，它允许在类、接口、方法中使用类型参数，以便在代码中使用通用的数据结构和算法，同时保持类型安全性。
    • 类型擦除（Type Erasure）：泛型在 Java 中通过类型擦除来实现。在编译时，泛型的类型信息被擦除，泛型类型参数被替换为它们的上界或Object。这是为了保持与Java之前版本的兼容性，因为JVM原生不支持泛型。类型擦除带来的一个结果是，在运行时无法获得泛型的具体类型参数，只能操作原始类型。
    • 常用的通配符：
      • ? 通配符：表示未知类型，用于在泛型方法或泛型类中传递不确定的类型。它可以代表任何具体类型。
      • extends 通配符：用于限制通配符的上界，表示通配符可以是某个类的子类或实现某个接口。例如：List<? extends Number> 表示一个元素为 Number 或其子类的列表。
      • super通配符：用于限制通配符的下界，表示通配符可以是某个类的父类。例如：List<? super Integer> 表示一个元素为 Integer 或其父类的列表。

  12. 内部类了解吗？匿名内部类了解吗？

    https://www.runoob.com/w3cnote/java-inner-class-intro.html

    • 内部类（Inner Class） 是定义在另一个类内部的类。内部类可以访问外部类的成员（包括私有成员），并且可以起到封装、组织和隐藏代码的作用。内部类分为以下几种类型：
      1. 成员内部类：定义在外部类的成员位置，可以访问外部类的成员。使用外部类的实例来创建内部类的实例。
      2. 局部内部类：定义在方法或作用域内部的类，只在定义它的方法或作用域中可见。通常用于解决特定问题的封装。
      3. 静态内部类：定义在外部类内部，但使用 static 修饰。它类似于外部类的静态成员，不依赖于外部类的实例。
    • 匿名内部类（Anonymous Inner Class） 是一种特殊类型的内部类，它没有明确的类名，通常用于在需要一个类的实例的地方创建临时的、单次使用的对象。匿名内部类通常是在创建接口的实例或继承抽象类的子类时使用，用来实现单个方法或提供定制的实现。

  13. BIO,NIO,AIO 有什么区别?

    • BIO 属于同步阻塞 IO 模型。同步阻塞 IO 模型中，应用程序发起 read 调用后，会一直阻塞，直到内核把数据拷贝到用户空间。
    • NIO属于同步非阻塞模型。 IO 多路复用模型中，线程首先发起 select 调用，询问内核数据是否准备就绪，等内核把数据准备好了，用户线程再发起 read 调用。read 调用的过程（数据从内核空间 -> 用户空间）还是阻塞的。
    • AIO 也就是 NIO 2。Java 7 中引入了 NIO 的改进版 NIO 2,它是异步 IO 模型。异步 IO 是基于事件和回调机制实现的，也就是应用操作之后会直接返回，不会堵塞在那里，当后台处理完成，操作系统会通知相应的线程进行后续的操作。

Java 集合框架

  1. 说说 List,Set,Map 三者的区别？三者底层的数据结构？

    List、Set 和 Map 是Java集合框架中的三种常见接口，用于存储和操作一组元素。它们有不同的特性和用途。

    List：

    • List 是有序集合，允许重复元素。
    • 可以根据索引访问元素，索引从0开始。
    • 常见的实现类包括 ArrayList、LinkedList 和 Vector。
    • 底层数据结构可以是Object[]数组（ArrayList）或双向链表（LinkedList）。

    Set：

    • Set 是无序集合，不允许重复元素。
    • 不支持索引访问元素。
    • 常见的实现类包括 HashSet、LinkedHashSet 和 TreeSet。
    • 底层采用 HashMap 来保存元素。

    Map：

    • Map 是键值对的映射，每个键对应一个值，键不允许重复。
    • 键是唯一的，值可以重复。
    • 常见的实现类包括 HashMap、LinkedHashMap 和 TreeMap。
    • 底层数据结构是由数组+链表组成的，当链表长度大于阈值8时，如果数组长度大于64链表会转化为红黑树，否则直接扩容。

  2. 有哪些集合是线程不安全的？怎么解决呢？

      • ArrayList：在多线程环境下，如果多个线程同时对 ArrayList 进行添加、删除等操作，可能导致不一致性或异常。

      • HashSet：同样，多线程同时修改 HashSet 可能会导致不可预测的结果。

      • HashMap：多线程同时修改 HashMap 可能会导致死循环、数据丢失等问题。

    • 为了解决多线程环境下的线程安全问题，你可以采取以下几种方式：

      • 使用线程安全的集合类：Java提供了一些线程安全的集合类，如 Vector、Hashtable、ConcurrentHashMap、CopyOnWriteArrayList、CopyOnWriteArraySet 等。这些集合类在设计上考虑了多线程并发访问的问题，可以安全地在多线程环境中使用。
      • 手动同步（使用锁）：你可以使用 synchronized 关键字或其他同步机制来保护非线程安全的集合访问。例如，使用 Collections.synchronizedList(new ArrayList<>()) 可以得到一个线程安全的 List 对象。
      • 使用并发集合类：Java 5引入了一些专门设计用于多线程环境的并发集合类，如 ConcurrentHashMap、ConcurrentLinkedQueue、ConcurrentSkipListSet 等。这些集合类在并发性能上做了优化，可以支持高并发访问。

  3. 比较 HashSet、LinkedHashSet 和 TreeSet 三者的异同

    • HashSet、LinkedHashSet 和 TreeSet 都是 Set 接口的实现类，都能保证元素唯一，并且都不是线程安全的。
    • HashSet、LinkedHashSet 和 TreeSet 的主要区别在于底层数据结构不同。HashSet 的底层数据结构是哈希表（基于 HashMap 实现）。LinkedHashSet 的底层数据结构是链表和哈希表，元素的插入和取出顺序满足 FIFO。TreeSet 底层数据结构是红黑树，元素是有序的，排序的方式有自然排序和定制排序。
    • 底层数据结构不同又导致这三者的应用场景不同。HashSet 用于不需要保证元素插入和取出顺序的场景，LinkedHashSet 用于保证元素的插入和取出顺序满足 FIFO 的场景，TreeSet 用于支持对元素自定义排序规则的场景。

  4. HashMap 和 Hashtable 的区别？HashMap 和 HashSet 区别？HashMap 和 TreeMap 区别？

      • 线程是否安全： HashMap 是非线程安全的，Hashtable 是线程安全的,因为 Hashtable 内部的方法基本都经过synchronized 修饰。

      • 效率： 因为线程安全的问题，HashMap 要比 Hashtable 效率高。

      • 对 Null key 和 Null value 的支持： HashMap 可以存储 null 的 key 和 value，但 null 作为键只能有一个，null 作为值可以有多个；Hashtable 不允许有 null 键和 null 值，否则会抛出 NullPointerException。

      • 底层数据结构： JDK1.8 以后的 HashMap 在解决哈希冲突时有了较大的变化，当链表长度大于阈值（默认为 8）时，将链表转化为红黑树（将链表转换成红黑树前会判断，如果当前数组的长度小于 64，那么会选择先进行数组扩容，而不是转换为红黑树），以减少搜索时间（后文中我会结合源码对这一过程进行分析）。Hashtable 没有这样的机制。

  5. HashMap 的底层实现

    • HashMap 是Java集合框架中的一个哈希表实现，用于存储键值对。底层基于数组+链表/红黑树。
    • 数组存储桶：HashMap 内部维护一个数组，称为存储桶（buckets），用于存储元素。每个桶可以容纳一个链表或红黑树。数组的大小会随着元素的增加而进行动态调整（扩容）。
    • 哈希码计算：当将键值对放入 HashMap 中时，首先会经过扰动函数处理过后得到 hash 值，然后通过 (n - 1) & hash 判断当前元素存放的位置（这里的 n 指的是数组的长度）。
    • 解决哈希冲突：由于不同的键可能产生相同的哈希码，可能会导致哈希冲突。通过拉链法解决冲突，HashMap 使用链表或红黑树来存储在同一个桶中的冲突元素。
    • 链表和红黑树：如果同一个桶中的链表长度较长（超过阈值8），如果此时数组长度大于64，那么HashMap 将链表转换为红黑树，否则进行扩容、重新哈希。
    • 扩容和重新哈希：当 HashMap 中的元素数量达到一定阈值时，会进行扩容，即创建一个更大的数组，并将现有元素重新哈希分布到新的数组中，以减少哈希冲突。

  6. HashMap 的长度为什么是 2 的幂次方

    • 为了能让 HashMap 存取高效，尽量较少碰撞，也就是要尽量把数据分配均匀。Hash 值的范围值较大，还要先做对数组的长度取模运算，得到的余数才能用来要存放的位置也就是对应的数组下标。

    • 我们首先可能会想到采用 % 取余的操作来实现，但实际计算公式是 (n - 1) & hash ，这是因为，取余(%)操作中如果除数是 2 的幂次则等价于与其除数减一的与(&)操作（也就是说 hash%length==hash&(length-1)的前提是 length 是 2 的 n 次方）。并且采用二进制位操作 &，相对于%能够提高运算效率。

    • 将哈希码的低位与长度进行取模，从而让元素更均匀地分布到不同的桶中。这有助于减少哈希冲突的概率，提高查找效率。

    • 实现线程安全的方式（重要）：
      • 在 JDK1.7 的时候，ConcurrentHashMap 对整个桶数组进行了分割分段(Segment，分段锁)，每一把锁只锁容器其中一部分数据（下面有示意图），多线程访问容器里不同数据段的数据，就不会存在锁竞争，提高并发访问率。
      • 到了 JDK1.8 的时候，ConcurrentHashMap 已经摒弃了 Segment 的概念，而是直接用 Node 数组+链表+红黑树的数据结构来实现，并发控制使用 synchronized 和 CAS 来操作。整个看起来就像是优化过且线程安全的 HashMap，虽然在 JDK1.8 中还能看到 Segment 的数据结构，但是已经简化了属性，只是为了兼容旧版本；
      • Hashtable(同一把锁) :使用 synchronized 来保证线程安全，效率非常低下。当一个线程访问同步方法时，其他线程也访问同步方法，可能会进入阻塞或轮询状态，如使用 put 添加元素，另一个线程不能使用 put 添加元素，也不能使用 get，竞争会越来越激烈效率越低。

  8. ConcurrentHashMap 线程安全的具体实现方式/底层具体实现

    • JDK1.7，首先将数据分为一段一段（这个“段”就是 Segment）的存储，然后给每一段数据配一把锁，当一个线程占用锁访问其中一个段数据时，其他段的数据也能被其他线程访问。Segment 的结构和 HashMap 类似，是一种数组和链表结构，一个 Segment 包含一个 HashEntry 数组，每个 HashEntry 是一个链表结构的元素，每个 Segment 守护着一个 HashEntry 数组里的元素，当对 HashEntry 数组的数据进行修改时，必须首先获得对应的 Segment 的锁。也就是说，对同一 Segment 的并发写入会被阻塞，不同 Segment 的写入是可以并发执行的。

    • JDK1.8，ConcurrentHashMap 取消了 Segment 分段锁，采用 Node + CAS + synchronized 来保证并发安全。数据结构跟 HashMap 1.8 的结构类似，数组+链表/红黑二叉树。Java 8 中，锁粒度更细，synchronized 只锁定当前链表或红黑二叉树的首节点，这样只要 hash 不冲突，就不会产生并发，就不会影响其他 Node 的读写，效率大幅提升。

JVM

  1. JVM 内存结构

    JVM内存结构分为运行时数据区域和本地内存

    • 运行时数据区域分为
      • 线程共享的堆，堆中有字符串常量池，字符串常量池主要是为了避免字符串的重复创建专门开辟的一块区域。
      • 线程私有的程序计数器、虚拟机栈和本地方法栈，程序计数器是一块很小的内存区域，可以看作是当前线程所执行的字节码的行号指示器；虚拟机栈是JVM运行时数据区域的核心，出了本地方法以外的所有Java方法都借助栈来实现，每一个方法调用都对应有一个栈帧被压入栈中，每个栈帧中都拥有：局部变量表、操作数栈、动态链接、方法返回地址；本地方法栈和虚拟机栈类似，但执行的是本地方法。
    • 本地内存有直接内存、元空间，元空间有运行时常量池，直接内存通过JNI方法在本地内存上分配；元空间和方法区是永久代的具体实现，主要存储一些类信息、字段信息、方法信息等；运行时常量池类似于传统编程语言的符号表。

  3. 什么是类加载？何时类加载？类加载流程？

    • 类加载是指将 Java 类的字节码文件加载到 Java 虚拟机中，并在内存中创建一个对应的 Class 对象的过程。
      • 显式加载：通过 Class.forName("className") 或者类的静态方法引用加载类。
      • 创建对象：当创建一个类的实例时，会触发该类的加载。
      • 访问静态变量/方法：当访问一个类的静态变量或方法时，会触发该类的加载。
      • 继承父类：当一个类继承自另一个类时，会触发父类和子类的加载。
      • 实现接口：当一个类实现接口时，会触发该接口的加载。
      • 加载（Loading）
        • 通过全类名获取定义此类的二进制字节流。
        • 将字节流所代表的静态存储结构转换为方法区的运行时数据结构。
        • 在内存中生成一个代表该类的 Class 对象，作为方法区这些数据的访问入口。
      • 链接（Linking）：链接阶段分为三个子阶段：验证、准备和解析。
        • 验证（Verification）：对字节码进行验证，确保符合虚拟机规范和安全要求。
        • 准备（Preparation）：准备阶段是正式为类变量分配内存并设置类变量初始值的阶段。
        • 解析（Resolution）：将符号引用（例如方法调用、字段引用等）转化为直接引用。
      • 初始化（Initialization）：在这个阶段，类的静态变量会被赋予初始值，静态块会被执行。这是类加载的最后一个阶段，也是真正开始运行 Java 程序的阶段。

  4. 知道哪些类加载器。类加载器之间的关系？

      • **BootstrapClassLoader(启动类加载器)**：最顶层的加载类，由 C++实现，通常表示为 null，并且没有父级，主要用来加载 JDK 内部的核心类库（ %JAVA_HOME%/lib目录下的 rt.jar、resources.jar、charsets.jar等 jar 包和类）以及被 -Xbootclasspath参数指定的路径下的所有类。
      • **ExtensionClassLoader(扩展类加载器)**：主要负责加载 %JRE_HOME%/lib/ext 目录下的 jar 包和类以及被 java.ext.dirs 系统变量所指定的路径下的所有类。
      • **AppClassLoader(应用程序类加载器)**：面向我们用户的加载器，负责加载当前应用 classpath 下的所有 jar 包和类。
      • CustomeClassLoader(自定义类加载器）：由应用程序开发者编写的类加载器，用于加载特定的类；可以继承自 java.lang.ClassLoader，重写 findClass 方法来实现类加载逻辑；用于实现类隔离、热部署、动态加载等功能。

    • 这些类加载器之间的关系是一个树状结构，自底向上依次是：启动类加载器 -> 扩展类加载器 -> 应用类加载器 -> 自定义类加载器。每个类加载器在加载类时，会首先尝试委托给其父加载器加载，如果父加载器无法加载，才会自行尝试加载。这种委托机制使得 Java 类加载器能够实现类隔离和类重用，同时也确保了类的一致性和正确性。

    • 自底向上判断类是否被加载，自顶向下加载类。

  5. 类加载器的双亲委派了解么？ 结合 Tomcat 说一下双亲委派（Tomcat 如何打破双亲委托机制？…）。

    • 双亲委派机制是 Java 类加载器的一种设计原则，它通过在加载类时首先委派给父类加载器加载，确保了类的一致性和避免了重复加载。其基本思想是，当一个类加载器收到加载请求时，它会首先检查自己是否已经加载了该类。如果没有，它会将加载请求委派给其父加载器。父加载器会依次尝试加载，如果父加载器也没有加载该类，就会继续向上委派，直到到达启动类加载器。如果所有父加载器都无法加载该类，才由当前加载器自行尝试加载。

    • :robot: Tomcat 作为一个 Java Web 服务器，也遵循了双亲委派机制。Tomcat 在启动时会创建多个类加载器，每个 Web 应用程序都有一个独立的类加载器。Tomcat 的类加载器层次结构大致如下：

      • Bootstrap Class Loader：Java 虚拟机的启动类加载器。
      • Catalina Class Loader：Tomcat 的共享类加载器，加载 Tomcat 自身的类库。
      • Webapp Class Loader：每个 Web 应用程序都有一个独立的类加载器，用于加载应用程序的类。

      我们比较熟悉的 Tomcat 服务器为了能够优先加载 Web 应用目录下的类，然后再加载其他目录下的类，就自定义了类加载器 WebAppClassLoader 来打破双亲委托机制。这也是 Tomcat 下 Web 应用之间的类实现隔离的具体原理。

  6. 为什么需要双亲委派

    • 双亲委派模型保证了 Java 程序的稳定运行，可以避免类的重复加载，也保证了 Java 的核心 API 不被篡改。

    • 如果没有使用双亲委派模型，而是每个类加载器加载自己的话就会出现一些问题，比如我们编写一个称为 java.lang.Object 类的话，那么程序运行的时候，系统就会出现两个不同的 Object 类。双亲委派模型可以保证加载的是 JRE 里的那个 Object 类，而不是你写的 Object 类。这是因为 AppClassLoader 在加载你的 Object 类时，会委托给 ExtClassLoader 去加载，而 ExtClassLoader 又会委托给 BootstrapClassLoader，BootstrapClassLoader 发现自己已经加载过了 Object 类，会直接返回，不会去加载你写的 Object 类。

  7. Java 内存模型

    • Java 内存模型（Java Memory Model，JMM）是一种规范，用于定义多线程程序中各个线程之间如何访问共享变量以及内存操作的顺序和效果。JMM 的设计目标是保证多线程程序在不同平台上都能表现出一致的行为，从而避免由于不同硬件和操作系统的差异而导致的并发问题。
    • JMM 的关键概念和规则包括：
      • 主内存和工作内存：
        • 主内存是所有线程共享的内存区域，存储共享变量。
        • 每个线程有自己的工作内存，存储主内存中的部分变量的副本。
      • 内存可见性：JMM 保证共享变量的值在主内存和工作内存之间的读写具有可见性，一个线程修改了共享变量的值，其他线程能够立即看到。
      • 原子性：JMM 保证对于简单操作（如读取、写入）的读写具有原子性，不会发生中间状态的读写操作。
      • 有序性：JMM 定义了一些规则来确保指令重排序不会影响多线程程序的正确性。
      • happens-before 关系：在多线程环境下定义了操作之间的偏序关系，用于指定某些操作的执行顺序。
      • volatile 关键字：volatile 关键字可以用来保证变量的可见性和禁止指令重排序。
      • synchronized 关键字：synchronized 关键字用于实现原子性和有序性。
      • final 关键字：final 关键字用于确保在构造函数完成之前，不会被其他线程看到半初始化的对象。

  8. 栈中存放什么数据，堆中呢？

    • 栈存放的数据：

      • 局部变量表：主要存放了编译期可知的各种数据类型（boolean、byte、char、short、int、float、long、double）、对象引用（reference 类型，它不同于对象本身，可能是一个指向对象起始地址的引用指针，也可能是指向一个代表对象的句柄或其他与此对象相关的位置）。
      • 操作数栈：主要作为方法调用的中转站使用，用于存放方法执行过程中产生的中间计算结果。另外，计算过程中产生的临时变量也会放在操作数栈中。
      • 动态链接：主要服务一个方法需要调用其他方法的场景。
      • 方法返回值

    • 堆（Heap）存放的数据：

      • 对象实例：所有通过 new 关键字创建的对象实例都存放在堆中。
      • 数组：数组也是对象，存放在堆中。

  9. 大对象放在哪个内存区域

    大对象就是需要大量连续内存空间的对象（比如：字符串、数组）。大对象直接进入老年代的行为是由虚拟机动态决定的，它与具体使用的垃圾回收器和相关参数有关。大对象直接进入老年代是一种优化策略，旨在避免将大对象放入新生代，从而减少新生代的垃圾回收频率和成本。

  10. 堆区如何分类

    在Java中，堆区可以根据对象的生命周期和分配方式进行分类。主要的分类包括：

    • 新生代（Young Generation）：
      • 新生代是堆区的一部分，用于存放新创建的对象。
      • 又分为 Eden区和两个 Survivor区（From区和To区）。
      • 大部分对象在新生代中被创建和销毁，存活时间较短。
    • 老年代（Old Generation / Tenured Generation）：
      • 老年代用于存放已经存活一段时间的对象，通常是从新生代晋升过来的对象。
      • 对象在老年代中存活时间较长，垃圾回收频率较低。
    • 永久代（Permanent Generation）（JDK 1.7之前）或元空间（Metaspace）（JDK 1.8及以后）：
      • 永久代（或元空间）用于存放类的元数据、静态变量、常量池等。
      • 永久代在JDK 1.8及以后被元空间所取代，元空间使用本地内存，而不是虚拟机内存。

  11. 垃圾回收有哪些算法

    • 标记-清除算法（Mark-Sweep）：首先标记出所有不需要回收的对象，在标记完成后统一回收掉所有没有被标记的对象。
      • 该算法分为两个阶段：标记和清除。
      • 在标记阶段，从根节点（如栈中的引用、静态变量等）出发，遍历对象图，标记出所有可达的对象。
      • 在清除阶段，对堆中的对象进行遍历，清除未被标记的对象。
      • 缺点：会产生内存碎片，分配较大对象时可能无法找到足够的连续空间。
    • 复制算法（Copying）：将内存分为大小相同的两块，每次使用其中的一块。当这一块的内存使用完后，就将还存活的对象复制到另一块去，然后再把使用的空间一次清理掉。这样就使每次的内存回收都是对内存区间的一半进行回收。
      • 将堆分为两个区域：From区和To区，每次只使用其中一个区域。
      • 在发生垃圾回收时，将存活的对象从From区复制到To区，然后清除From区中的所有对象。
      • 适用于存活对象较少的情况，效率较高，但需要额外的空间。
    • 标记-整理算法（Mark-Compact）：根据老年代的特点提出的一种标记算法，标记过程仍然与“标记-清除”算法一样，但后续步骤不是直接对可回收对象回收，而是让所有存活的对象向一端移动，然后直接清理掉端边界以外的内存。
    • 分代收集算法：
      • 基于对象的生命周期将堆分为不同的代（新生代、老年代等）。
      • 新生代通常采用复制算法，老年代采用标记-整理算法。

  12. GC 的全流程

    • 标记（Marking）：
      • 从根节点出发，标记出所有可达的对象。根节点包括：虚拟机栈(栈帧中的本地变量表)中引用的对象、本地方法栈(Native 方法)中引用的对象、方法区中类静态属性引用的对象、方法区中常量引用的对象、所有被同步锁持有的对象、JNI（Java Native Interface）引用的对象
      • 通过遍历对象图，将可达对象标记为存活对象。
    • 清除（Sweeping）：
      • 在标记阶段后，进行清除操作。清除未被标记的对象。
      • 清除操作可以通过遍历堆中的所有对象，如果发现对象未被标记，则将其从内存中删除。
    • 整理（Compacting）（部分算法需要）：
      • 对内存进行整理操作，将存活的对象移动到一端，以解决内存碎片问题。
      • 整理后，内存空间会更加紧凑，减少了存储分配操作的开销。

  13. GC 中老年代用什么回收方法？

    老年代的对象存活几率是比较高的，而且没有额外的空间对它进行分配担保，所以我们必须选择“标记-清除”或“标记-整理”算法进行垃圾收集。

多线程

  1. 线程和进程的区别。

    • 一个进程中可以有多个线程，多个线程共享进程的堆和方法区 (JDK1.8 之后的元空间)资源，但是每个线程有自己的程序计数器、虚拟机栈 和 本地方法栈。

    • 线程是进程划分成的更小的运行单位。线程和进程最大的不同在于基本上各进程是独立的，而各线程则不一定，因为同一进程中的线程极有可能会相互影响。线程执行开销小，但不利于资源的管理和保护（并发问题）；而进程正相反。

  2. 什么是线程死锁?如何避免死锁?

    • 多个线程同时被阻塞，它们中的一个或者全部都在等待某个资源被释放。由于线程被无限期地阻塞，因此程序不可能正常终止。
    • 死锁的四个必要条件：
      • 互斥条件：该资源任意一个时刻只由一个线程占用。
      • 请求与保持条件：一个线程因请求资源而阻塞时，对已获得的资源保持不放。
      • 不剥夺条件:线程已获得的资源在未使用完之前不能被其他线程强行剥夺，只有自己使用完毕后才释放资源。
      • 循环等待条件:若干线程之间形成一种头尾相接的循环等待资源关系。
    • 避免死锁就是在资源分配时，借助于算法（比如银行家算法）对资源分配进行计算评估，使其进入安全状态。

  3. 乐观锁和悲观锁了解么？

    • 悲观锁总是假设最坏的情况，认为共享资源每次被访问的时候就会出现问题(比如共享数据被修改)，所以每次在获取资源操作的时候都会上锁，这样其他线程想拿到这个资源就会阻塞直到锁被上一个持有者释放。也就是说，共享资源每次只给一个线程使用，其它线程阻塞，用完后再把资源转让给其它线程。
    • 乐观锁总是假设最好的情况，认为共享资源每次被访问的时候不会出现问题，线程可以不停地执行，无需加锁也无需等待，只是在提交修改的时候去验证对应的资源（也就是数据）是否被其它线程修改了（具体方法可以使用版本号机制或 CAS 算法）。

  4. 说说 sleep() 方法和 wait() 方法区别和共同点?

    • 共同点：两者都可以暂停线程的执行。
    • 区别：
      • sleep() 方法没有释放锁，而 wait() 方法释放了锁 。
      • wait() 通常被用于线程间交互/通信，sleep()通常被用于暂停执行。
      • wait() 方法被调用后，线程不会自动苏醒，需要别的线程调用同一个对象上的 notify()或者 notifyAll() 方法。sleep()方法执行完成后，线程会自动苏醒，或者也可以使用 wait(long timeout) 超时后线程会自动苏醒。
      • sleep() 是 Thread 类的静态本地方法，wait() 则是 Object 类的本地方法。为什么这样设计呢？下一个问题就会聊到

  5. Java 线程池有哪些参数？阻塞队列有几种？拒绝策略有几种？新线程添加的流程?

    • 线程池参数

      • corePoolSize : 任务队列未达到队列容量时，最大可以同时运行的线程数量。

      • maximumPoolSize : 任务队列中存放的任务达到队列容量的时候，当前可以同时运行的线程数量变为最大线程数。

      • workQueue: 新任务来的时候会先判断当前运行的线程数量是否达到核心线程数，如果达到的话，新任务就会被存放在队列中。

      • keepAliveTime:线程池中的线程数量大于 corePoolSize 的时候，如果这时没有新的任务提交，核心线程外的线程不会立即销毁，而是会等待，直到等待的时间超过了 keepAliveTime才会被回收销毁；

      • unit : keepAliveTime 参数的时间单位。

      • threadFactory :executor 创建新线程的时候会用到。

      • handler :饱和策略。关于饱和策略下面单独介绍一下。

    • 阻塞队列

      • 容量为 Integer.MAX_VALUE 的 LinkedBlockingQueue（无界队列）：FixedThreadPool 和 SingleThreadExector 。由于队列永远不会被放满，因此FixedThreadPool最多只能创建核心线程数的线程。
      • SynchronousQueue（同步队列）：CachedThreadPool 。SynchronousQueue 没有容量，不存储元素，目的是保证对于提交的任务，如果有空闲线程，则使用空闲线程来处理；否则新建一个线程来处理任务。也就是说，CachedThreadPool 的最大线程数是 Integer.MAX_VALUE ，可以理解为线程数是可以无限扩展的，可能会创建大量线程，从而导致 OOM。
      • DelayedWorkQueue（延迟阻塞队列）：ScheduledThreadPool 和 SingleThreadScheduledExecutor 。DelayedWorkQueue 的内部元素并不是按照放入的时间排序，而是会按照延迟的时间长短对任务进行排序，内部采用的是“堆”的数据结构，可以保证每次出队的任务都是当前队列中执行时间最靠前的。DelayedWorkQueue 添加元素满了之后会自动扩容原来容量的 1/2，即永远不会阻塞，最大扩容可达 Integer.MAX_VALUE，所以最多只能创建核心线程数的线程。

    • 拒绝策略

      • ThreadPoolExecutor.AbortPolicy： 抛出 RejectedExecutionException来拒绝新任务的处理。
      • ThreadPoolExecutor.CallerRunsPolicy： 调用执行自己的线程运行任务，也就是直接在调用execute方法的线程中运行(run)被拒绝的任务，如果执行程序已关闭，则会丢弃该任务。因此这种策略会降低对于新任务提交速度，影响程序的整体性能。如果您的应用程序可以承受此延迟并且你要求任何一个任务请求都要被执行的话，你可以选择这个策略。
      • ThreadPoolExecutor.DiscardPolicy： 不处理新任务，直接丢弃掉。
      • ThreadPoolExecutor.DiscardOldestPolicy： 此策略将丢弃最早的未处理的任务请求。

    • 流程

      • 如果当前运行的线程数小于核心线程数，那么就会新建一个线程来执行任务。
      • 如果当前运行的线程数等于或大于核心线程数，但是小于最大线程数，那么就把该任务放入到任务队列里等待执行。
      • 如果向任务队列投放任务失败（任务队列已经满了），但是当前运行的线程数是小于最大线程数的，就新建一个线程来执行任务。
      • 如果当前运行的线程数已经等同于最大线程数了，新建线程将会使当前运行的线程超出最大线程数，那么当前任务会被拒绝，饱和策略会调用RejectedExecutionHandler.rejectedExecution()方法。

  6. 实现 Runnable 接口和 Callable 接口的区别。

    • 返回值类型：

      • Runnable接口的run方法没有返回值，因此线程执行完任务后无法返回任何结果。
      • Callable接口的call方法可以返回一个结果，这个结果的类型可以是任何类型，包括基本数据类型、对象引用等。

    • 异常处理：

      • Runnable接口的run方法不能抛出受检查异常，只能抛出未受检查异常（RuntimeException）。
      • Callable接口的call方法可以抛出受检查异常，因此可以更灵活地处理异常情况。

    • 使用方式：

      • Runnable通常用于创建无返回值的线程，用于执行一些任务，但不需要返回结果。
      • Callable通常用于创建有返回值的线程，可以获取线程执行完任务后的结果，适用于需要计算结果的任务。

    • Future对象：

      • 当使用Callable创建线程时，可以获得一个Future对象，用于获取线程执行结果、取消线程等操作。
      • 使用Runnable时，通常不会获得这样的Future对象。

    • 兼容性：

      • Runnable接口是早期的多线程机制，在Java 1.0就引入了。
      • Callable接口是在Java 5.0引入的，它更灵活，支持有返回值和受检查异常。

    总结来说，如果你需要创建一个简单的线程来执行某些任务而不需要返回结果，Runnable接口足够了。但如果你需要线程执行完任务后返回一个结果，并且需要处理受检查异常，那么应该使用Callable接口。Callable通常与ExecutorService一起使用，以方便管理线程池和获取线程执行结果。

  7. 讲一下 JMM(Java 内存模型)。volatile 关键字解决了什么问题？说说 synchronized 关键字和 volatile 关键字的区别。

    • JMM前文有提到，这里不再赘述。
    • volatile关键字解决的问题
      • 可见性：在 Java 中，volatile 关键字可以保证变量的可见性，如果我们将变量声明为 volatile ，这就指示 JVM，这个变量是共享且不稳定的，每次使用它都到主存中进行读取。
      • 禁止指令重排序：在 Java 中，volatile 关键字除了可以保证变量的可见性，还有一个重要的作用就是防止 JVM 的指令重排序。如果我们将变量声明为 volatile ，在对这个变量进行读写操作的时候，会通过插入特定的 内存屏障 的方式来禁止指令重排序。
    • synchronized 关键字和 volatile 关键字是两个互补的存在，而不是对立的存在！
      • volatile 关键字是线程同步的轻量级实现，所以 volatile性能肯定比synchronized关键字要好 。但是 volatile 关键字只能用于变量而 synchronized 关键字可以修饰方法以及代码块 。
      • volatile 关键字能保证数据的可见性，但不能保证数据的原子性。synchronized 关键字两者都能保证。
      • volatile关键字主要用于解决变量在多个线程之间的可见性，而 synchronized 关键字解决的是多个线程之间访问资源的同步性

  8. AQS 原理了解么？AQS 组件有哪些？

    • AQS 核心思想是，如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的工作线程，并且将共享资源设置为锁定状态。如果被请求的共享资源被占用，那么就需要一套线程阻塞等待以及被唤醒时锁分配的机制，这个机制 AQS 是基于 CLH 锁 （Craig, Landin, and Hagersten locks） 实现的。
      • ReentrantLock：可重入锁的实现。
      • Semaphore：信号量的实现。
      • CountDownLatch：倒计数器的实现。
      • CyclicBarrier：循环屏障的实现。
      • ReentrantReadWriteLock：可重入读写锁的实现。

  9. 用过 CountDownLatch 么？什么场景下用的？

    • CountDownLatch 允许 count 个线程阻塞在一个地方，直至所有线程的任务都执行完毕。CountDownLatch 是一次性的，计数器的值只能在构造方法中初始化一次，之后没有任何机制再次对其设置值，当 CountDownLatch 使用完毕后，它不能再次被使用。
      • 某一线程在开始运行前等待 n 个线程执行完毕 : 将 CountDownLatch 的计数器初始化为 n ，每当一个任务线程执行完毕，就将计数器减 1 （countdownlatch.countDown()），当计数器的值变为 0 时，在 CountDownLatch 上 await() 的线程就会被唤醒。一个典型应用场景就是启动一个服务时，主线程需要等待多个组件加载完毕，之后再继续执行。
      • 实现多个线程开始执行任务的最大并行性：注意是并行性，不是并发，强调的是多个线程在某一时刻同时开始执行。类似于赛跑，将多个线程放到起点，等待发令枪响，然后同时开跑。做法是初始化一个共享的 CountDownLatch 对象，将其计数器初始化为 1 （new CountDownLatch(1)），多个线程在开始执行任务前首先 coundownlatch.await()，当主线程调用 countDown() 时，计数器变为 0，多个线程同时被唤醒。

MySQL基础

  1. 非关系型数据库和关系型数据库的区别？

    非关系型数据库（NoSQL数据库）和关系型数据库（SQL数据库）是两种不同的数据库管理系统，它们在数据存储和查询方面有许多区别。以下是它们的主要区别：

    • 数据模型：

      • 关系型数据库：使用表格（表）来存储数据，这些表之间通过外键关联。数据必须具有固定的结构，且需要定义表的模式（模式通常用SQL语言定义）。
      • 非关系型数据库：使用多种数据模型，如文档、键值、列族、图等。这些模型允许存储半结构化或无结构化数据，不需要预先定义模式。

    • 数据一致性：

      • 关系型数据库：通常遵循ACID（原子性、一致性、隔离性、持久性）事务，确保数据的完整性和一致性。
      • 非关系型数据库：通常使用BASE（基本可用、软状态、最终一致性）模型，允许一定程度的数据不一致性，但提高了可用性和性能。

    • 扩展性：

      • 关系型数据库：通常需要垂直扩展（增加更强大的硬件）或分区表来处理高负载，扩展性有限。
      • 非关系型数据库：通常设计为横向扩展，可以通过添加更多节点来处理大规模数据和高负载，具有更好的扩展性。

    • 查询语言：

      • 关系型数据库：使用结构化查询语言（SQL）来执行复杂的查询操作。
      • 非关系型数据库：通常使用不同的查询语言或API，具体取决于数据库类型。

    • 适用场景：

      • 关系型数据库：适用于事务处理（如金融应用、在线购物）和需要复杂查询的应用。
      • 非关系型数据库：适用于大数据、实时分析、内容管理、社交媒体、日志记录等应用，其中数据结构可能会频繁更改。

    • 灵活性：

      • 关系型数据库：通常需要提前定义数据模型，难以应对数据结构的频繁变化。
      • 非关系型数据库：更灵活，能够适应数据模型的变化。

    • 复制和分片：

      • 关系型数据库：复制和分片通常需要复杂的配置和管理。
      • 非关系型数据库：通常具有内置的复制和分片功能，更容易扩展和管理。

    总之，选择关系型数据库还是非关系型数据库取决于应用程序的需求和特点。通常情况下，关系型数据库适合具有严格一致性要求的应用，而非关系型数据库适用于需要处理大量非结构化数据、需要高可用性和可扩展性的应用。很多情况下，应用程序会使用多种数据库类型来满足不同需求。

  2. 事务的四大特性了解么?

    • 原子性（Atomicity）：事务是最小的执行单位，不允许分割。事务的原子性确保动作要么全部完成，要么完全不起作用；
    • 一致性（Consistency）：执行事务前后，数据保持一致，例如转账业务中，无论事务是否成功，转账者和收款人的总额应该是不变的；
    • 隔离性（Isolation）：并发访问数据库时，一个用户的事务不被其他事务所干扰，各并发事务之间数据库是独立的；
    • 持久性（Durability）：一个事务被提交之后。它对数据库中数据的改变是持久的，即使数据库发生故障也不应该对其有任何影响。

  3. MySQL 事务隔离级别？默认是什么级别？

      • **READ-UNCOMMITTED(读取未提交)**：最低的隔离级别，允许读取尚未提交的数据变更，可能会导致脏读、幻读或不可重复读。
      • **READ-COMMITTED(读取已提交)**：允许读取并发事务已经提交的数据，可以阻止脏读，但是幻读或不可重复读仍有可能发生。
      • **REPEATABLE-READ(可重复读)**：对同一字段的多次读取结果都是一致的，除非数据是被本身事务自己所修改，可以阻止脏读和不可重复读，但幻读仍有可能发生。
      • **SERIALIZABLE(可串行化)**：最高的隔离级别，完全服从 ACID 的隔离级别。所有的事务依次逐个执行，这样事务之间就完全不可能产生干扰，也就是说，该级别可以防止脏读、不可重复读以及幻读。
    • MySQL InnoDB 存储引擎的默认支持的隔离级别是 REPEATABLE-READ（可重读）。

  4. 乐观锁与悲观锁的区别?

    MySQL 中并发事务的控制方式无非就两种：锁 和 MVCC。

    • 锁可以看作是悲观控制的模式。
    • 多版本并发控制（MVCC，Multiversion concurrency control）可以看作是乐观控制的模式。

  5. MySQL 数据库两种存储引擎的区别?

    • InnoDB 支持行级别的锁粒度，MyISAM 不支持，只支持表级别的锁粒度。

    • MyISAM 不提供事务支持。InnoDB 提供事务支持，实现了 SQL 标准定义了四个隔离级别。

    • MyISAM 不支持外键，而 InnoDB 支持。

    • MyISAM 不支持 MVCC，而 InnoDB 支持。

    • 虽然 MyISAM 引擎和 InnoDB 引擎都是使用 B+Tree 作为索引结构，但是两者的实现方式不太一样。

    • MyISAM 不支持数据库异常崩溃后的安全恢复，而 InnoDB 支持。

    • InnoDB 的性能比 MyISAM 更强大。

MySQL索引

  1. 为什么索引能提高查询速度?

    索引是一种用于快速查询和检索数据的数据结构，其本质可以看成是一种排序好的数据结构。

    • 快速定位数据：索引允许数据库引擎快速定位包含特定值的行，而不必扫描整个数据表。这是因为索引按照某种排序方式存储，使得数据库可以使用二分查找或类似的算法快速定位所需的数据行。

    • 减少数据扫描：在没有索引的情况下，执行查询通常需要扫描整个数据表以查找匹配条件的行。使用索引可以将数据表的扫描量大大减少，从而显著提高查询性能，尤其是对于大型数据表。

    • 加速排序和连接：索引不仅用于快速查找单个行，还可以用于加速排序操作和连接操作。例如，如果您需要按某一列排序结果，索引可以减少排序的时间。在连接多个表时，索引也可以加速连接操作。

    • 减少磁盘I/O：通过使用索引，数据库引擎可以更有效地利用磁盘读取操作，因为它可以跳过不相关的数据块，只读取包含所需数据的块。这减少了磁盘I/O的开销，提高了查询性能。

    • 提高并发性：索引可以减少数据的锁定时间，因为查询可以更快地定位所需的数据，从而允许其他事务更快地访问数据表中的其他部分。

  2. 聚集索引和非聚集索引的区别？非聚集索引一定回表查询吗?

      • 聚簇索引，即索引结构和数据一起存放的索引，并不是一种单独的索引类型。InnoDB 中的主键索引就属于聚簇索引。
      • 非聚簇索引，即索引结构和数据分开存放的索引，并不是一种单独的索引类型。二级索引(辅助索引)就属于非聚簇索引。MySQL 的 MyISAM 引擎，不管主键还是非主键，使用的都是非聚簇索引。

    • 当利用到覆盖索引时，非聚簇索引不会进行回表查询。下文会讲到什么是覆盖索引。

  3. 索引底层的数据结构了解么？Hash 索引和 B+树索引优劣分析

    • 索引底层的数据结构是 B+ Tree ,只有叶子结点保存存放 key 和 data，叶子结点构成双向循环链表便于顺序查询。
    • Hash索引
      • 优点：
        • 快速的查找：Hash索引提供了非常快速的查找性能。它通过将索引键值散列到一个固定大小的桶（bucket）中，以常量时间的复杂度来查找数据。
        • 适用于等值查找：Hash索引非常适合等值查找操作，例如精确的查找、等于某个值的查询等。这使得它在需要快速查找某个唯一值时非常有效。
      • 缺点：
        • 不适用于范围查询：Hash索引不支持范围查询，如大于、小于或区间查询。这是因为散列函数通常会将相似的键映射到不同的桶中，导致范围查询效率低下。
        • 不支持排序：Hash索引不支持排序，因为数据在桶内的排列顺序是不确定的。
        • 冲突问题：当多个不同的键值散列到同一个桶中，就会发生冲突。这需要解决冲突的策略，例如使用链式散列来存储冲突的值，这可能会导致性能下降。
    • B+树索引
      • 优点：
        • 支持范围查询和排序：B+树索引支持范围查询，例如大于、小于、区间查询等，并且可以用于排序操作，因为B+树中的数据是有序存储的。
        • 稳定性好：B+树索引在数据插入和删除时维护平衡，因此具有较好的稳定性。
      • 缺点：
        • 查找性能相对较慢：相对于Hash索引，B+树索引的查找性能稍慢，因为需要从根节点开始遍历树来定位数据。
        • 占用更多的存储空间：B+树索引通常占用更多的存储空间，因为需要存储树的结构信息。
    • Hash索引适用于需要快速查找特定值的场景，尤其是当你知道唯一键值时，它通常用于内存数据库或缓存中，以提供快速的数据检索；B+树索引适用于各种查询需求的场景，尤其是在需要范围查询、排序和模糊查询时，它通常用于磁盘数据库，因为它对磁盘I/O的效率更高。

  4. B+树做索引比红黑树好在哪里？

    红黑树的平衡性相对较弱，可能会导致树的高度较高，这可能会导致一些数据需要进行多次磁盘 IO 操作才能查询到，这也是 MySQL 没有选择红黑树的主要原因。

  5. 最左前缀匹配原则了解么？

    在使用联合索引时，MySQL 会根据联合索引中的字段顺序，从左到右依次到查询条件中去匹配，如果查询条件中存在与联合索引中最左侧字段相匹配的字段，则就会使用该字段过滤一批数据，直至联合索引中全部字段匹配完成，或者在执行过程中遇到范围查询（如 **>、<**）才会停止匹配。对于 >=、<=、BETWEEN、like 前缀匹配的范围查询，并不会停止匹配。所以，我们在使用联合索引时，可以将区分度高的字段放在最左边，这也可以过滤更多数据。

  6. 什么是覆盖索引

    • 如果一个索引包含（或者说覆盖）所有需要查询的字段的值，我们就称之为 覆盖索引（Covering Index） 。

    • 覆盖索引即需要查询的字段正好是索引中的字段，那么直接根据该索引，就可以查到数据了，而无需回表查询。

  3. 简单说一下 SQL 调优思路。

    • 避免使用 select*。
    • 当数据量较大的时候，普通的分页耗费时间较长，可以采用子查询和延迟关联。
    • 避免多表 join 操作。可以采用单表查询后在内存中做关联；或者采用数据冗余。
    • 不建议使用外键和级联，因为对分库分表不友好。
    • 选择合适的字段
      • 比如ip地址字符串可以转成整型数据来存储；
      • 对于自增id这种非负数的数据来说最好采用无符号整型来存储；
      • 小数值类型比如年龄、性别可以采用tinyint类型；
      • 日期类型采用timestamp，相比datetime 8 字节，timestamp 4 字节；
    • 数据库数据更新操作，能使用批量操作就使用批量操作，因为大量的单次更新只会无端的耗费网络资源。
    • 慢查询日志
    • 正确使用索引
      • 不为 NULL 的字段 ：索引字段的数据应该尽量不为 NULL，因为对于数据为 NULL 的字段，数据库较难优化。
      • 被频繁查询的字段 ：我们创建索引的字段应该是查询操作非常频繁的字段。
      • 被作为条件查询的字段 ：被作为 WHERE 条件查询的字段，应该被考虑建立索引。
      • 频繁需要排序的字段 ：索引已经排序，这样查询可以利用索引的排序，加快排序查询时间。
      • 被经常频繁用于连接的字段 ：经常用于连接的字段可能是一些外键列，对于外键列并不一定要建立外键，只是说该列涉及到表与表的关系。对于频繁被连接查询的字段，可以考虑建立索引，提高多表连接查询的效率。
      • 被频繁更新的字段不应该设置为索引，因为会耗费大量的资源去维护索引。
      • 尽可能建立联合索引而不是单列索引，不要设置冗余索引。
      • 多多利用前缀索引，比如在text字段，varchar字段上面建立前缀索引。

  4. 简单说一下大表优化的思路。

    :robot:

    • 索引优化：

      • 确保合适的索引存在，以加速常见查询。索引可以帮助数据库引擎快速定位数据行，避免全表扫描。但要小心不要过度索引，因为维护索引也会占用资源。
      • 对于大表，考虑使用覆盖索引，以减少IO操作，因为索引可以覆盖查询所需的列，而不必访问数据行。

    • 分区和分表：

      • 将大表拆分成更小的分区或分表，根据业务需求和查询模式来决定如何拆分。这可以减少单个表的数据量，提高查询性能。
      • 使用数据库系统提供的分区功能，如MySQL的分区表，可以更容易地管理分区数据。

    • 定期清理和归档：

      • 定期清理过期或不再需要的数据，以减少表格的大小。归档历史数据到归档表中，保留当前数据在主表中。
      • 考虑使用自动化作业来管理数据清理和归档过程。

    • 垂直切分：

      • 将大表中的列按照访问模式不同切分到不同的表中。这样可以减少单个表的宽度，提高查询性能。
      • 保持常用列在主表中，将不常用的列放在辅助表中，只在需要时进行联接查询。

    • 缓存和缓存策略：

      • 使用缓存技术，如缓存服务器或CDN，来缓存热门查询的结果，以减轻数据库负载。
      • 使用适当的缓存策略，如定时过期或基于LRU（最近最少使用）的缓存策略。

    • 硬件升级：

      • 考虑升级数据库服务器的硬件，包括CPU、内存和磁盘，以提高处理大表的性能。
      • 考虑使用SSD硬盘，因为它们通常比传统磁盘更快。

    • 查询优化：

      • 通过使用合适的查询语句和索引，优化查询，确保只检索所需的数据，而不是全表扫描。
      • 使用分页和限制结果集的方式来避免一次性检索大量数据。

    • 数据库引擎选择：

      • 考虑使用适合大表的数据库引擎，一些NoSQL数据库或分布式数据库可能更适合处理大规模数据。

    • 监控和性能调整：

      • 使用数据库性能监控工具来跟踪数据库性能，并根据需要进行调整。
      • 定期分析数据库的执行计划，确保查询优化。

    • 数据压缩：

      • 对于历史数据或稀疏数据，考虑使用数据压缩技术，以减小存储需求并提高查询性能。

  5. 分库分表了解么？为什么要分库分表？有哪些常见的分库分表工具(sharding-jdbc、TSharding、MyCAT…)？

    • 分库：是将数据库中的数据分散到不同的数据库上，可以垂直分库，也可以水平分库。
      • 垂直分库 就是把单一数据库按照业务进行划分，不同的业务使用不同的数据库，进而将一个数据库的压力分担到多个数据库。
      • 水平分库 是把同一个表按一定规则拆分到不同的数据库中，每个库可以位于不同的服务器上，这样就实现了水平扩展，解决了单表的存储和性能瓶颈的问题。
    • 分表：就是对单表的数据进行拆分，可以是垂直拆分，也可以是水平拆分。
      • 垂直分表 是对数据表列的拆分，把一张列比较多的表拆分为多张表。举个例子：我们可以将用户信息表中的一些列单独抽出来作为一个表。
      • 水平分表 是对数据表行的拆分，把一张行比较多的表拆分为多张表，可以解决单一表数据量过大的问题。举个例子：我们可以将用户信息表拆分成多个用户信息表，这样就可以避免单一表数据量过大对性能造成影响。
      • 单表的数据达到千万级别以上，数据库读写速度比较缓慢。
      • 数据库中的数据占用的空间越来越大，备份时间越来越长。
      • 应用的并发量太大。

  6. Redis 给缓存数据设置过期时间有啥用？ Redis 是如何判断数据是否过期的呢？

  10. 什么是 Sentinel？ 有什么用？

    • Sentinel（哨兵） 只是 Redis 的一种运行模式 ，不提供读写服务，依赖于 Redis 工作。Redis Sentinel 实现 Redis 集群高可用，只是在主从复制实现集群的基础下，多了一个 Sentinel 角色来帮助我们监控 Redis 节点的运行状态并自动实现故障转移。
      • 监控：监控所有 redis 节点（包括 sentinel 节点自身）的状态是否正常。
      • 故障转移：如果一个 master 出现故障，Sentinel 会帮助我们实现故障转移，自动将某一台 slave 升级为 master，确保整个 Redis 系统的可用性。
      • 通知 ：通知 slave 新的 master 连接信息，让它们执行 replicaof 成为新的 master 的 slave。
      • 配置提供 ：客户端连接 sentinel 请求 master 的地址，如果发生故障转移，sentinel 会通知新的 master 链接信息给客户端。

  11. Sentinel 如何检测节点是否下线？主观下线与客观下线的区别?

    • 每个 sentinel 节点以每秒钟一次的频率向整个集群中的 master、slave 以及其他 sentinel 节点发送一个 PING 命令。
      • 主观下线(SDOWN) ：sentinel 节点认为某个 Redis 节点已经下线了（主观下线），但还不是很确定，需要其他 sentinel 节点的投票。
      • 客观下线(ODOWN) ：法定数量（通常为过半）的 sentinel 节点认定某个 Redis 节点已经下线（客观下线），那它就算是真的下线了。

  12. Sentinel 是如何实现故障转移的？

    • sentinel 集群中选择一个 leader ，让 leader 来负责完成故障转移。
    • 第一步：在已下线主节点（旧主节点）属下的所有「从节点」里面，挑选出一个从节点，并将其转换为主节点。
    • 第二步：让已下线主节点属下的所有「从节点」修改复制目标，修改为复制「新主节点」；
    • 第三步：将新主节点的 IP 地址和信息，通过「发布者/订阅者机制」通知给客户端；
    • 第四步：继续监视旧主节点，当这个旧主节点重新上线时，将它设置为新主节点的从节点；

  13. Sentinel 如何选择出新的 master（选举机制）?

    • 首先把已经下线的结点过来吧，然后是把网络状态不好的过滤掉。

    • slave 优先级 ：可以通过修改 slave-priority 配置的值来手动设置 slave 的优先级。slave-priority默认值为 100，其值越小得分越高，越有机会成为 master。不过，0 是一个特殊的优先级值 ，如果一个 slave 的 slave-priority值为 0，代表其没有参加 master 选举的资格。如果没有优先级最高的，再判断复制进度。

    • 复制进度 ：Sentinel 总是希望选择出数据最完整也就是复制进度最快的 slave 被提升为新的 master，复制进度越快得分也就越高。

    • runid(运行 id) ：通常经过前面两轮筛选已经成功选出来了新的 master，万一真有多个 slave 的优先级和复制进度一样的话，那就 runid 小的成为新的 master，每个 redis 节点启动时都有一个 40 字节随机字符串作为运行 id。

  14. 如何从 Sentinel 集群中选择出 Leader ？

    分布式领域的 共识算法 ，让分布式系统中的节点就一个问题达成共识。在 sentinel 选举 leader 这个场景下，这些 sentinel 要达成的共识就是谁才是 leader 。

  15. Sentinel 可以防止脑裂吗？

    调整以下两个参数

    • min-replicas-to-write 1：用于配置写 master 至少写入的 slave 数量，设置为 0 表示关闭该功能。3 个节点的情况下，可以配置为 1 ，表示 master 必须写入至少 1 个 slave ，否则就停止接受新的写入命令请求。

  16. 为什么需要 Redis Cluster？解决了什么问题？有什么优势？

      • 缓存的数据量太大，比如成百上千G；并发量要求太大
      • Redis Cluster 通过 分片来进行数据管理，提供 主从复制、故障转移等开箱即用的功能，可以非常方便地帮助我们解决 Redis 大数据量缓存以及 Redis 服务高可用的问题。
      • Redis Cluster 这种方案可以很方便地进行 横向拓展 ，内置了开箱即用的解决方案。
      • 可以横向扩展缓解写压力和存储压力，支持动态扩容和缩容。
      • 具备主从复制、故障转移（内置了 Sentinel 机制，无需单独部署 Sentinel 集群）等开箱即用的功能。
      • 主从复制和 Redis Sentinel 这两种方案本质都是通过增加主库（master）的副本（slave）数量的方式来提高 Redis 服务的整体可用性和读吞吐量，都不支持横向扩展来缓解写压力以及解决缓存数据量过大的问题。

  17. Redis Cluster 是如何分片的？

    • Redis Cluster 并没有使用一致性哈希，采用的是 哈希槽分区 ，每一个键值对都属于一个 hash slot（哈希槽） 。

    • 创建并初始化 Redis Cluster 的时候，Redis 会自动平均分配这 16384 个哈希槽到各个节点，不需要我们手动分配。

    • 客户端连接 Redis Cluster 中任意一个 master 节点即可访问 Redis Cluster 的数据，当客户端发送命令请求的时候，需要先根据 key 通过上面的计算公示找到的对应的哈希槽，然后再查询哈希槽和节点的映射关系，即可找到目标节点。

  18. 为什么 Redis Cluster 的哈希槽是 16384 个?

    • 哈希槽太大会导致心跳包太大，消耗太多带宽；
    • 哈希槽总数越少，对存储哈希槽信息的 bitmap 压缩效果越好；
    • Redis Cluster 的主节点通常不会扩展太多，16384 个哈希槽已经足够用了。

    先通过公式 CRC16(key) mod 16384 计算 HASH_SLOT ，根据 HASH_SLOT 的值即可找到对应的哈希槽
